1. Kotlin Fundamentals

Kotlin is the table-stakes language. These ten cover what gets asked in every screen.

Q1Easy

What's the difference between val, var, and const val?

val = read-only reference (still calls a getter), var = mutable reference, const val = compile-time constant inlined at use sites.

val means the reference can't be reassigned, but the object it points to can still mutate (e.g., val list = mutableListOf<Int>() still allows list.add(1)). var means the reference itself can be reassigned. const val must be a primitive or String, must be top-level or in a companion object, and the value gets baked into the bytecode at every call site (no field read at runtime). 

val name: String = "Sai"           // can't reassign
var count: Int = 0                // can reassign
const val MAX = 100             // inlined at compile time
Follow-up: "When would you prefer const val over val?" Answer: when the value never changes and you want zero runtime overhead — typical for HTTP timeouts, magic numbers, BuildConfig keys.
Gotcha: val is not immutable; only the reference is. To get true immutability you also need an immutable type (e.g., List, data class with val properties).
Q2Easy

How does Kotlin null safety work? ? vs !! vs ?. vs ?:

Types ending in ? can hold null; the compiler forces you to handle the null case before calling members.
  • String? — nullable type. Can't call .length directly.
  • ?. — safe call. name?.length returns null if name is null instead of crashing.
  • ?: — Elvis operator. name?.length ?: 0 returns 0 if name is null.
  • !! — non-null assertion. name!!.length throws NullPointerException if null. Code smell — almost always the wrong tool.
  • let — runs a block only if non-null: name?.let { print(it.length) }.
val name: String? = getName()
val len = name?.length ?: 0     // safe + default
name?.let { send(it) }            // run only if non-null
Follow-up: "Why is !! a smell?" It defers a possible NPE to runtime, which is exactly what null safety is meant to prevent. If you truly know it's non-null, restructure or use requireNotNull with a clear message.

Q4Medium

What's a data class, what does it generate, and what are its limits?

A data class auto-generates equals, hashCode, toString, componentN (for destructuring), and copy based on its primary constructor's properties.
  • Equality is structural: two data class User("a") instances are equals.
  • Generated functions consider only properties declared in the primary constructor.
  • copy() lets you make modified copies: user.copy(age = 30).
  • Destructuring: val (id, name) = user.
  • Cannot be abstract, open, sealed, or inner.
  • Must have at least one parameter in the primary constructor.
data class User(val id: Int, val name: String)

val u1 = User(1, "Sai")
val u2 = u1.copy(name = "Kalyan")
val (id, name) = u1                // destructuring
Follow-up: "What if I add a property in the class body, not the constructor?" It's not used in equals/hashCode/toString — surprising bug source.
Gotcha: Don't put mutable properties (var) in a data class — hashCode can change after insertion into a HashSet, breaking lookups.
Q5Medium

What's a sealed class? When do you use it over enum?

A sealed class restricts inheritance to a fixed, compile-time-known set of subclasses — perfect for modeling exhaustive state and result types where each variant carries different data.

The compiler knows every subtype, so when on a sealed type is exhaustive without an else branch. Unlike enum, each subtype can be a separate class with its own properties and behavior. 

sealed class UiState<out T> {
    object Loading : UiState<Nothing>()
    data class Success<T>(val data: T) : UiState<T>()
    data class Error(val msg: String) : UiState<Nothing>()
}

when (state) {                          // exhaustive — no else needed
    is UiState.Loading -> spinner()
    is UiState.Success -> render(state.data)
    is UiState.Error   -> showError(state.msg)
}

Use enum only when all variants share the same shape (just labels).

Follow-up: "What's sealed interface?" Same idea, but works as an interface so you can implement multiple — useful for cross-cutting concerns. Added in Kotlin 1.5.
Q6Medium

What's a higher-order function? What does inline do?

A higher-order function takes or returns a function. inline tells the compiler to copy the lambda body into the call site, eliminating the lambda's object allocation and enabling non-local returns.

Every Kotlin lambda compiles to a Function object — meaning a small allocation per call. Marking the higher-order function inline removes that overhead, which matters in hot loops. 

inline fun <T> measure(block: () -> T): T {
    val t = System.nanoTime()
    return block().also { print(System.nanoTime() - t) }
}

// At the call site, the body is copied in — no Function object created.
val result = measure { compute() }
  • Use inline when the function takes a lambda and is called in hot paths.
  • Use noinline on a specific lambda parameter to opt out (e.g., when storing it).
  • Use crossinline to disallow non-local returns from the lambda.
Gotcha: Don't inline functions with large bodies — bytecode bloats and may slow things down. Inlining is for small wrappers, not 200-line functions.
Q7Medium

Extension functions — how do they work under the hood?

Extension functions are syntactic sugar — the compiler converts them into static functions taking the receiver as their first parameter. They're resolved statically, not virtually.

They don't actually add methods to the class; you can't override an extension function and you can't call extensions on subtypes via polymorphism. 

fun String.lastChar(): Char = get(length - 1)
"hello".lastChar()
// compiles to:
// public static char lastChar(String $receiver) { ... }
// StringExtKt.lastChar("hello");
Follow-up: "What if the class has a member with the same signature?" The member always wins. Extensions extend, they don't override.
Q8Medium

What's the difference between object, class, and companion object?

class = blueprint, object = singleton instance (declaration), companion object = a singleton tied to a containing class (the only way to write static-like members in Kotlin).
class Logger { fun log(s: String) { ... } }
val l = Logger()                  // new instance

object AppPrefs {                 // thread-safe singleton, lazy init
    var theme: String = "dark"
}
AppPrefs.theme = "light"           // no instantiation

class User(val name: String) {
    companion object {
        fun fromJson(json: String): User = /* ... */
    }
}
User.fromJson("...")              // looks like static

object declarations are initialized lazily on first access and are thread-safe by default (the JVM class loader guarantees it).

Follow-up: "Is object a true singleton across classloaders?" In Android, yes (one classloader per app). In multi-classloader scenarios (rare), each classloader has its own instance.

Q10Hard

Generics: covariance (out), contravariance (in), and what's a reified type parameter?

out T = produces T (subtype safe in), in T = consumes T (supertype safe in), reified = type info preserved at runtime via inlining.

JVM generics are erased at runtime — by default you can't write T::class. reified with inline tells the compiler to substitute the actual type at each call site. 

// Covariance — Producer<Cat> is a Producer<Animal>
interface Producer<out T> { fun produce(): T }

// Contravariance — Consumer<Animal> is a Consumer<Cat>
interface Consumer<in T> { fun consume(t: T) }

// reified — type usable at runtime, only inside inline functions
inline fun <reified T> Bundle.parcelable(key: String): T? =
    if (Build.VERSION.SDK_INT >= 33) getParcelable(key, T::class.java)
    else @Suppress("DEPRECATION") getParcelable(key) as? T

val user: User? = bundle.parcelable("user")
Follow-up: "Why does reified require inline?" Because the type substitution happens by copying the function body into the call site — only inline functions get copied.

2. Coroutines & Flow

This is where modern Android lives. Expect 3-5 questions from this section in any senior screen.

Q11Easy

What's a coroutine, and how does it differ from a thread?

A coroutine is a suspendable computation — it can pause without blocking the underlying thread. Threads are OS-scheduled and expensive (~1 MB stack each); coroutines are cheap (~hundreds of bytes) and many run on a small thread pool.

Inside a suspend function, calling another suspend function may pause execution. The compiler converts that into a state machine: each suspension point becomes a state, and resumption simply jumps to the next state — no thread is held while waiting. 

suspend fun loadUser(): User {
    val json = api.fetch()        // suspends, doesn't block
    return json.toUser()
}

scope.launch {                    // fire-and-forget coroutine
    val user = loadUser()
    showUser(user)
}

You can spawn 100k coroutines in 100 MB; spawning 100k threads will OOM your phone before the loop finishes.

Follow-up: "Then how does a coroutine actually wait for I/O?" The blocking call moves to a thread pool (Dispatchers.IO) — the coroutine itself just suspends until the I/O completes.
Q12Easy

What are Dispatchers.Main, IO, Default, and Unconfined?

Dispatchers map coroutines onto threads. Main = UI thread, Default = CPU work (cores-sized pool), IO = blocking I/O (64+ threads, expandable), Unconfined = run wherever the resumption happens (rarely useful).
  • Main: Android's UI thread. Touch events, View updates, Compose recomposition.
  • Default: backed by a thread pool sized to max(2, cpuCores). Use for CPU-bound work — JSON parsing, image processing, sorting.
  • IO: starts at 64 threads, expands if needed. Use for blocking I/O — Room queries, file reads, OkHttp synchronous calls.
  • Main.immediate: Skips dispatch if you're already on Main — faster than Main.
  • Unconfined: testing only.
withContext(Dispatchers.IO) { db.query(...) }    // switch dispatcher for one block
viewModelScope.launch(Dispatchers.Default) { sort(list) }
Gotcha: Dispatchers.Default and IO share the same underlying thread pool — switching between them is cheap.
Q13Medium

What's structured concurrency? Why does it matter?

Every coroutine has a parent scope; cancelling the parent cancels all children automatically. No leaked work, no orphaned coroutines, no manual bookkeeping.

Pre-coroutines, async libraries leaked work routinely — fire a network call from onCreate, the user backs out, the call still completes and tries to update a destroyed view. Coroutines fix this: 

class MyViewModel : ViewModel() {
    init {
        viewModelScope.launch {        // child of viewModelScope
            val user = api.fetch()
            _state.value = user
        }
    }
}
// when ViewModel.onCleared() runs, viewModelScope is cancelled,
// all children (including the launch above) get cancelled too.

Common scopes: viewModelScope, lifecycleScope, rememberCoroutineScope() in Compose.

Follow-up: "What's SupervisorJob?" A Job variant where one child's failure doesn't cancel siblings — useful in scopes hosting independent tasks (e.g., Repository fetching three things in parallel where one failing shouldn't kill the others).
Q14Medium

launch vs async — when to use which?

launch = fire-and-forget, returns Job. async = need a result, returns Deferred<T> you can await(). Use async only for parallel decomposition.
scope.launch { saveAnalytics(event) }     // don't care about result

scope.launch {
    val user = async { api.user() }       // parallel
    val orders = async { api.orders() }
    render(user.await(), orders.await())   // joined
}

Don't use async { ... }.await() back-to-back — that's just sequential suspend calls with extra overhead. The point of async is parallelism.

Gotcha: Exceptions in async are deferred until await(). If you forget to await, the exception is silently swallowed (in a SupervisorJob scope) or crashes the app (in a Job scope) — surprising either way.
Q15Medium

How does coroutine cancellation work? Why is it cooperative?

Cancellation throws CancellationException at the next suspension point. Tight CPU loops won't see it unless they call yield() or ensureActive().
scope.launch {
    while (isActive) {            // check periodically
        heavyWork()
        yield()                   // or ensureActive()
    }
}

// Wrong — no suspension point, can't be cancelled:
scope.launch { while (true) { computeForever() } }

All suspend functions in kotlinx.coroutines (delay, withContext, await, etc.) check cancellation. Custom suspend wrappers around callbacks should call ensureActive() to be safe.

Follow-up: "How do I run cleanup that survives cancellation?" Wrap it in withContext(NonCancellable) { ... }. Use sparingly — usually for releasing resources.
Q16Medium

Exception handling in coroutines: try/catch vs CoroutineExceptionHandler

Synchronous-style: wrap suspending code in try/catch. For top-level uncaught exceptions in a launch, install a CoroutineExceptionHandler on the scope or root job.
val handler = CoroutineExceptionHandler { _, e ->
    Log.e("App", "Uncaught", e)
}

val scope = CoroutineScope(SupervisorJob() + Dispatchers.Main + handler)

scope.launch {
    try { api.call() }
    catch (e: IOException) { showError() }    // IOExc handled here
                                              // other exceptions go to handler
}
  • CancellationException is special — never logged as an error.
  • async + await propagates the exception at await(); try/catch around await.
  • In a non-supervised scope, child failure cancels the entire scope.
Gotcha: CoroutineExceptionHandler does NOT receive exceptions from async — those propagate via await().

Q18Hard

StateFlow vs SharedFlow vs LiveData — which to use?

StateFlow = current value + subsequent values (UI state). SharedFlow = configurable replay/buffer (events). LiveData = lifecycle-aware, Java/Compose-friendly, but Coroutines/Flow are now preferred.

val state = MutableStateFlow(UiState.Loading)

// Events (toasts, snackbars, navigation)
val events = MutableSharedFlow<Event>(replay = 0, extraBufferCapacity = 1)

// Collect from UI safely
lifecycleScope.launch {

Q19Hard

Why stateIn(WhileSubscribed(5000)) instead of Eagerly or Lazily?

5000 ms gives a grace period across config changes (rotation, system bar dismissal) without restarting upstream work, but still cancels when the user truly leaves.
val users: StateFlow<List<User>> = repository.observeUsers()
    .stateIn(
        scope = viewModelScope,
        started = SharingStarted.WhileSubscribed(5_000),
        initialValue = emptyList()
    )
  • Eagerly: starts immediately, never stops. Wastes battery if UI not visible.
  • Lazily: starts on first subscriber, never stops. Same waste post-first-collect.
  • WhileSubscribed(5000): starts on first sub, stops 5 s after last sub. Bridges rotation. Recommended for UI flows.
Follow-up: "Why specifically 5000?" Empirical — rotation/config-change typically completes < 1 s; 5 s also covers brief background navigation. Adjust if your app has longer transitions.
Q20Hard

Flow operators: map vs flatMapConcat vs flatMapMerge vs flatMapLatest

map = 1:1 transform. The flatMap variants flatten Flow<Flow<T>> → Flow<T>, differing in whether they wait, run in parallel, or cancel previous.
  • flatMapConcat: serialize — wait for inner Flow to complete before starting next.
  • flatMapMerge: parallel — start all inner Flows, merge emissions.
  • flatMapLatest: cancel previous inner Flow when a new one arrives. Most common for search.
// Search-as-you-type — cancel old query when new one arrives
queryFlow
    .debounce(300)
    .distinctUntilChanged()
    .flatMapLatest { q -> api.searchFlow(q) }
    .collect { results -> render(results) }
Follow-up: "Difference between combine and zip?" combine emits whenever any source emits (using latest of others); zip waits for all to emit one value each, like a strict pair.
Q21Hard

How do you turn a callback API into a Flow? (callbackFlow)

callbackFlow {} registers the callback at the start, calls trySend for each event, and uses awaitClose {} to unregister when the collector is gone.
fun locationUpdates(client: FusedLocationProviderClient): Flow<Location> = callbackFlow {
    val request = LocationRequest.create().apply {
        priority = Priority.PRIORITY_HIGH_ACCURACY
        interval = 5_000
    }
    val cb = object : LocationCallback() {
        override fun onLocationResult(r: LocationResult) {
            r.lastLocation?.let { trySend(it) }
        }
    }
    client.requestLocationUpdates(request, cb, Looper.getMainLooper())
    awaitClose { client.removeLocationUpdates(cb) }
}

awaitClose is mandatory — the Flow won't compile without it. It guarantees you unregister the listener when the collector is cancelled.

Follow-up: "What's the difference between trySend and send?" send suspends if the channel buffer is full; trySend returns immediately and tells you if it was accepted. For high-frequency callbacks (sensor updates), trySend with a CONFLATED buffer is ideal.
Q22Hard

What's Channel, and how does it differ from a Flow?

Channel = hot communication primitive between coroutines (queue with suspending send/receive). Flow = a description of a stream that you collect to start. Use Channel for cross-coroutine communication; use Flow for transformations and UI state.
val ch = Channel<Event>(Channel.BUFFERED)

scope.launch { for (e in ch) handle(e) }       // receiver
ch.send(Event.Click)                              // suspending
ch.trySend(Event.Click).isSuccess                 // non-suspending
ch.close()                                        // no more sends

Channel options: RENDEZVOUS (no buffer, hand-off), BUFFERED (default 64), UNLIMITED, CONFLATED (drop old).

Follow-up: "Why prefer SharedFlow over Channel for events?" SharedFlow supports multiple collectors and is integrated with the Flow ecosystem. Channel is point-to-point. For UI events with one collector and back-pressure, both work; SharedFlow tends to be more idiomatic now.

3. Android Lifecycle & Components

If you can't explain process death and ViewModel survival, you're a Junior on paper.

Q23Easy

Activity lifecycle in order — and what runs on a rotation?

onCreate → onStart → onResume ↔ onPause → onStop → onDestroy. On rotation: onPause → onStop → onSaveInstanceState → onDestroy → onCreate(savedInstance) → onStart → onRestoreInstanceState → onResume.
  • onCreate: setup that survives config changes (binding inflation, ViewModel init).
  • onStart: visible to user.
  • onResume: in foreground, has focus.
  • onPause: losing focus (dialog, multi-window). Save lightweight state here, but be quick.
  • onStop: no longer visible. Free heavyweight resources.
  • onDestroy: about to be killed (config change OR user-initiated finish).

Use isFinishing or isChangingConfigurations in onDestroy to tell the two apart.

Gotcha: Don't write to disk in onPause — short window, can ANR. Use onStop for that, or async via WorkManager.
Q24Medium

How does ViewModel survive configuration changes?

ViewModelStore is held by the Activity's NonConfigurationInstance. The store survives the destroy/create cycle of rotation; it's only cleared when the Activity is finishing for real.

The Android framework retains the NonConfigurationInstance across config-change destroy/create. Jetpack stuffs the ViewModelStore in there. ViewModelProvider reads from the same store on the new Activity instance, returning the same ViewModel. 

class MyViewModel(val repo: Repo) : ViewModel() {
    val data: StateFlow<...> = repo.observe()
        .stateIn(viewModelScope, WhileSubscribed(5_000), null)

    override fun onCleared() {
        // fired only when truly destroyed, not on rotation
    }
}
Follow-up: "Does ViewModel survive process death?" No. Use SavedStateHandle for the small state that needs to survive that.
Q25Medium

What's process death and how do you handle it?

Process death = Android killed your backgrounded app to reclaim memory. On relaunch your stack of Activities is restored, but in-memory state is gone. Use SavedStateHandle for transient UI state and Room/DataStore for everything else.
class EditViewModel(
    private val handle: SavedStateHandle
) : ViewModel() {
    val draft: StateFlow<String> = handle.getStateFlow("draft", "")

    fun setDraft(text: String) { handle["draft"] = text }
}

Test it by enabling Developer Options → "Don't keep activities", or via adb shell am kill com.your.app. Most production bugs hide here.

Gotcha: SavedStateHandle goes through Bundle — values must be Parcelable, ≤ 1 MB total. For larger state, persist to Room.
Q26Medium

Activity vs Fragment vs Compose screen — which to use today?

2026 default: a single Activity hosting Compose. Fragments are still fine for legacy XML codebases or when you need framework Fragment APIs (e.g., Material's BottomSheetDialogFragment, ViewPager2 with FragmentStateAdapter).
  • Activity: one per app for most modern apps. Holds NavGraph, theme, system bars.
  • Fragment: optional. Compose Navigation can replace most use cases.
  • Compose screen: a top-level @Composable function. Holds its own ViewModel via viewModel().

The big wins from single-Activity: cleaner deep-linking, simpler shared element transitions, easier theme/state sharing.

Follow-up: "Why not zero Activities?" The OS needs at least one entry point — that's the launch Activity in your manifest.
Q27Medium

What are lifecycleScope, repeatOnLifecycle, and why use them?

lifecycleScope ties a coroutine to the Activity/Fragment lifecycle (cancelled at onDestroy). repeatOnLifecycle(STARTED) additionally cancels and restarts the block as the lifecycle moves above/below STARTED.
// Anti-pattern: collects even when invisible — wastes battery
lifecycleScope.launch { vm.state.collect { render(it) } }

// Correct: only collects while STARTED
lifecycleScope.launch {
    repeatOnLifecycle(Lifecycle.State.STARTED) {
        vm.state.collect { render(it) }
    }
}

In Compose, this is automatic via collectAsStateWithLifecycle().

Gotcha: Plain collect from a Flow inside a Fragment without repeatOnLifecycle continues collecting in background — drains battery and may try to update detached views.
Q28Medium

What's the difference between onSaveInstanceState and ViewModel?

Bundle (onSaveInstanceState) survives config change AND process death — but is small (≤ 1 MB total system-wide) and Parcelable-only. ViewModel survives config change but NOT process death — but holds anything in memory.

Q29Hard

Activity launch modes — standard, singleTop, singleTask, singleInstance

Standard = always new instance, on top of caller. SingleTop = reuse if already on top. SingleTask = single instance per task, brought to front. SingleInstance = its own task, no other Activities.
  • standard (default): every startActivity creates a new instance. Most screens.
  • singleTop: if same Activity is already at the top of the stack, deliver as onNewIntent; don't re-create. Use for search results, notifications.
  • singleTask: one instance per task. Re-launching brings it to front; everything above it gets cleared. Use for app entry / launcher.
  • singleInstance: one instance, alone in its own task. Rare — used by some system apps (calls, alarms).

You can also pass intent flags at launch: FLAG_ACTIVITY_NEW_TASK, FLAG_ACTIVITY_CLEAR_TOP, etc.

Gotcha: singleTask/singleInstance break startActivityForResult. Single Activity + Compose Navigation sidesteps this whole topic.
Q30Hard

What's a Service? Foreground vs Bound vs Started — and what's required in 2026?

Service = component that runs without a UI. Started = fire-and-forget. Bound = client-server (IBinder). Foreground = persistent notification + can run in background. Android 14+ requires foregroundServiceType in the manifest.
  • Started: startService(). Mostly replaced by WorkManager. Killable when backgrounded.
  • Bound: clients bindService(); communicate via IBinder/AIDL. Survives as long as a client is bound.
  • Foreground: startForegroundService() + within 5-10 s call startForeground() with a notification. Required for ongoing user-aware tasks (music, GPS during trip, file upload).
// AndroidManifest.xml — Android 14+ requires this:
<service
    android:name=".MyMusicService"
    android:foregroundServiceType="mediaPlayback"
    android:exported="false" />

Valid types: camera, microphone, location, mediaPlayback, mediaProjection, health, connectedDevice, dataSync, etc. Misdeclaring crashes the service on 14+.

Follow-up: "When is WorkManager wrong?" When you need real-time ongoing work the user is aware of (music, navigation, broadcast). WorkManager is for deferrable, guaranteed work.

4. Jetpack Compose

Compose dominates 2026. Expect 3-5 questions; some interviewers focus their entire screen here.

Q31Easy

What is Jetpack Compose, and how does it differ from XML/View?

Compose is a declarative, Kotlin-DSL UI toolkit. You describe what the UI should look like for a given state; Compose figures out how to update it. No findViewById, no XML, no manual mutation.
  • Declarative: UI = f(state). Same state → same UI. Pure function.
  • Recomposition: when state changes, only affected Composables re-run.
  • No view hierarchy backing: Compose has its own slot table; widgets aren't Views (until interop with AndroidView).
  • Fewer bugs: state and UI can't drift out of sync.
@Composable
fun Greeting(name: String) {
    Text("Hello, $name!")
}
Follow-up: "Can I mix Compose with existing Views?" Yes. Use ComposeView in XML, or AndroidView inside Compose to host a View. Migrations are typically incremental.
Q32Easy

What's recomposition? When does it happen?

Recomposition = re-running a Composable function with new state. Triggered when a State<T> the Composable reads changes value.

Compose tracks which Composable read which state. When that state's value changes, Compose schedules that Composable (and only descendants that depend on the change) for recomposition. 

@Composable
fun Counter() {
    var count by remember { mutableStateOf(0) }
    Button(onClick = { count++ }) {        // updates count
        Text("Clicks: $count")            // reads count → recomposes
    }
}

Recomposition is idempotent: running the same Composable with the same inputs must produce the same UI, every time. Don't perform side effects directly inside a Composable body.

Gotcha: Reading a value from a non-State source (plain var) won't cause recomposition. Only State<T> reads are tracked.
Q33Medium

remember vs rememberSaveable — what's the difference?

remember survives recomposition; rememberSaveable additionally survives configuration changes and process death (via Bundle).
var ephemeralCount by remember { mutableStateOf(0) }
var persistedCount by rememberSaveable { mutableStateOf(0) }
  • rememberSaveable uses Bundle under the hood — value must be Parcelable, Serializable, or have a custom Saver.
  • Use rememberSaveable for UI state the user expects to persist across rotation (text field input, scroll position handled by rememberLazyListState).
  • Use plain remember for derived/cached values that can be recomputed.
Follow-up: "What if I want to remember a custom class?" Provide a Saver<T, Bundle> or use mapSaver / listSaver.
Q34Medium

What is state hoisting?

Move state out of a Composable to its caller; the Composable becomes stateless and receives (value, onValueChange) as parameters. This makes it reusable, testable, and previewable.
// Bad: state lives inside, can't be controlled or shared
@Composable
fun NameInput() {
    var name by remember { mutableStateOf("") }
    TextField(name, onValueChange = { name = it })
}

// Good: hoisted
@Composable
fun NameInput(name: String, onNameChange: (String) -> Unit) {
    TextField(name, onValueChange = onNameChange)
}

// Caller owns the state
var name by rememberSaveable { mutableStateOf("") }
NameInput(name, onNameChange = { name = it })

Rule of thumb: hoist state to the lowest common ancestor of all Composables that need it.

Follow-up: "When should state live in a ViewModel vs hoisted in Compose?" UI-only state (focus, expanded/collapsed, scroll position) → Compose. Business state, async results → ViewModel.
Q35Medium

Side effects in Compose: LaunchedEffect, DisposableEffect, SideEffect, rememberCoroutineScope

Composables must be pure. Side effects (network calls, listeners, timers) live in side-effect APIs that are tied to the Composable's lifecycle.
  • LaunchedEffect(key): runs a coroutine; cancels and restarts if key changes.
  • DisposableEffect(key): runs setup; onDispose for teardown. Use to register/unregister listeners.
  • SideEffect: runs after every successful recomposition. Use to publish to non-Compose code.
  • rememberCoroutineScope(): returns a CoroutineScope tied to the Composable; launch from event handlers (clicks, drags).
@Composable
fun UserScreen(userId: String) {
    LaunchedEffect(userId) {                  // re-runs on userId change
        viewModel.load(userId)
    }
    val scope = rememberCoroutineScope()
    Button(onClick = {
        scope.launch { viewModel.refresh() }   // from a click
    }) { Text("Refresh") }

    DisposableEffect(Unit) {
        analytics.startTracking()
        onDispose { analytics.stopTracking() }
    }
}
Gotcha: Never call LaunchedEffect from inside an onClick — it's a Composable, not a function. Use rememberCoroutineScope for event-driven launches.
Q36Medium

What's derivedStateOf? When do you need it?

Wrap an expensive computed value derived from other State so it only recomputes (and triggers recomposition) when the result actually changes — not when every input changes.
// Bad: recomposes on EVERY scroll pixel — even when scrollPos hasn't crossed 100
val showButton = listState.firstVisibleItemIndex > 10

// Good: only recomposes when the boolean actually changes
val showButton by remember {
    derivedStateOf { listState.firstVisibleItemIndex > 10 }
}

Without derivedStateOf, every change to firstVisibleItemIndex (which changes every scroll tick) recomposes the reader. With it, only the transition across the threshold matters.

Follow-up: "When is it overkill?" When the input changes infrequently — adding derivedStateOf adds an allocation and an extra layer for no gain.
Q37Medium

What does key (or itemKey) do in LazyColumn?

Tells Compose how to identify items across data changes so it can preserve state and animate moves correctly. Without keys, Compose uses positional identity — bad when items are added/removed.
LazyColumn {
    items(
        items = users,
        key = { it.id }                       // stable identity
    ) { user ->
        UserRow(user)
    }
}

Without keys, removing the first item makes Compose think every other item changed — destroying internal state and re-running animations. Always provide a stable key (typically the entity ID).

Gotcha: Don't use index as the key — defeats the purpose. Don't use the entire object — equality must be cheap.
Q38Hard

What's Compose stability? @Stable vs @Immutable — and why does it matter?

"Stable" types let Compose skip recomposition when inputs are equal. If a parameter type isn't stable, Compose conservatively recomposes every time the parent does.
  • Compose treats primitives, Strings, and types it can prove are stable (data classes with stable fields, lambdas without unstable captures) as stable.
  • List<T>, Map<K,V> from the standard library are unstable (they could be MutableList).
  • @Immutable: promise to Compose this type's properties never change after construction.
  • @Stable: properties may change, but you'll notify Compose (via State).
@Immutable
data class UiState(
    val users: ImmutableList<User>,           // kotlinx.collections.immutable
    val isLoading: Boolean
)

In Compose Compiler 1.5.4+ (and Compose 1.6+ "strong skipping"), Compose can skip recomposition for unstable parameters too if they're equals(). Still, marking types @Immutable is best practice.

Follow-up: "How do I check stability?" Add the Compose Compiler metrics flag — it generates a report listing every Composable's stability and where it's broken.
Q39Hard

Compose performance: what causes unnecessary recompositions and how to find them?

Causes: unstable parameters, lambdas re-created each frame, reading State too high in the tree, computing derived values without derivedStateOf. Find them via the Layout Inspector's "Recomposition Counts" or the Compose Compiler metrics report.
  • Layout Inspector → Recomposition Counts: shows which Composables recompose how often.
  • Compose Compiler reports: per-Composable stability + skippable info.
  • Defer state reads: pass a lambda { scrollState.value } instead of the value itself, so only the reader recomposes.
  • Hoist state up only as needed; don't keep it in a parent that doesn't read it.
  • Use movableContentOf when reusing UI under different parents.
// Bad: every recomposition makes a new lambda → child sees "different" callback
ChildButton(onClick = { vm.click() })

// Better: stable function reference
val onClick = remember { { vm.click() } }
ChildButton(onClick = onClick)
Gotcha: Don't run a profiler in debug builds. Always profile a release build with R8 + Baseline Profile installed — debug Compose performance is misleading.
Q40Hard

How do you collect a Flow safely in Compose?

Use collectAsStateWithLifecycle() from the lifecycle-compose artifact — it suspends collection when the screen isn't STARTED, like repeatOnLifecycle for Compose.
@Composable
fun Screen(vm: MyViewModel) {
    val state by vm.uiState.collectAsStateWithLifecycle()
    // vs the older collectAsState() which keeps collecting in background
    render(state)
}
  • collectAsState: collects regardless of lifecycle — drains battery in background.
  • collectAsStateWithLifecycle: lifecycle-aware. Default in 2026.
Follow-up: "What about LiveData?" Use livedata.observeAsState() — also lifecycle-aware via Compose's lifecycle integration.
Q41Hard

How does Compose Navigation work? When use it vs Fragment Navigation?

Compose Navigation 2.8+ uses type-safe routes via Kotlin serialization — pass data class objects as destinations, not strings. Use Compose Nav for Compose-only screens; Fragment Nav for hybrid XML/Compose codebases.
@Serializable data object Home
@Serializable data class Profile(val userId: String)

NavHost(navController, startDestination = Home) {
    composable<Home> { HomeScreen(navController) }
    composable<Profile> { entry ->
        val args = entry.toRoute<Profile>()
        ProfileScreen(args.userId)
    }
}

navController.navigate(Profile(userId = "42"))

Type-safe Nav eliminates a whole class of runtime "missing argument" bugs from the string-based era.

Gotcha: Don't pass complex domain objects through nav arguments — pass IDs, fetch in destination's ViewModel. Bundle has size limits.
Q42Hard

What's a Modifier, why does order matter, and what's Modifier.Node?

A Modifier is an immutable, ordered chain of behaviors applied to a Composable (padding, click, background). Order matters because each modifier wraps the next. Modifier.Node is the new (2023+) low-allocation, lifecycle-aware API for writing custom modifiers.
// 16 dp padding, then background → padding outside the color
Box(Modifier.padding(16.dp).background(Color.Red))

// background, then padding → padding inside the color
Box(Modifier.background(Color.Red).padding(16.dp))

Modifier.Node replaces the older composed { } approach. Nodes are reusable across recompositions — fewer allocations, better performance for custom modifiers. 

// Custom modifier with Modifier.Node — recommended in 2026
class RippleNode : Modifier.Node(), DrawModifierNode { ... }
fun Modifier.ripple() = this then RippleElement
Follow-up: "Why is order so important?" Each modifier produces a new wrapped layout. Padding-then-background paints behind the padded area; background-then-padding paints behind the unpadded area.

5. Architecture & Dependency Injection

How you structure code = level signal. These are about MVVM/MVI, Repository, Hilt, scopes.

Q43Easy

What is MVVM and why is it the default in Android?

MVVM = Model · ViewModel · View. View observes ViewModel state and dispatches user intents; ViewModel orchestrates the Model (Repository) and exposes UI state. Pairs perfectly with the framework's lifecycle ViewModel + Compose's reactive UI.
  • View (Activity/Fragment/Composable): renders state, forwards events. No business logic.
  • ViewModel: holds StateFlow<UiState>, exposes intents (functions). No Android framework references.
  • Model: Repository + data sources (Room, Retrofit). No UI knowledge.
UI ──intent──> ViewModel ──> Repository ──> (Room | Network)
UI <──state── ViewModel <── Repository <── (Room | Network)

The unidirectional data flow makes state predictable and testable.

Follow-up: "Why not MVP or MVC?" Both required manual lifecycle handling and were prone to view leaks. ViewModel-as-class survives configuration changes for free.
Q44Medium

MVVM vs MVI — what's the difference?

MVI is MVVM with two stricter rules: (1) UI state is a single immutable object, (2) all changes happen via a single reduce(state, intent) → state function. Classic state-machine architecture.
data class UiState(val users: List<User> = emptyList(), val loading: Boolean = false)

sealed interface Intent {
    data object Refresh : Intent
    data class Select(val id: String) : Intent
}

fun reduce(s: UiState, i: Intent): UiState = when (i) {
    Intent.Refresh    -> s.copy(loading = true)
    is Intent.Select -> s.copy(...)
}

Pros: explicit, testable, time-travel debugging. Cons: more boilerplate. Most teams pick MVVM unless the screen has > 5-6 distinct states.

Follow-up: "When does MVI shine?" Complex async screens with many edge cases (chat, video player, multi-step forms). The explicit state machine catches missed transitions at compile time.
Q45Medium

Why Repository pattern? What problem does it solve?

It hides the data sources from the rest of the app. ViewModel asks the Repo for users; the Repo decides whether to read from Room, Network, or both — and the ViewModel doesn't care.
interface UserRepository {
    fun observeUsers(): Flow<List<User>>
    suspend fun refresh()
}

class UserRepositoryImpl(
    private val dao: UserDao,
    private val api: UserApi
) : UserRepository {
    override fun observeUsers() = dao.observeAll()
    override suspend fun refresh() {
        val remote = api.fetch()
        dao.upsertAll(remote)
    }
}
  • Test seam: provide a fake repo in unit tests.
  • Single source of truth: UI always reads from Room (offline support automatic).
  • Swap data source without touching ViewModel.
Gotcha: Don't expose Room entities directly — map to domain models in the Repo. Otherwise schema changes leak everywhere.
Q46Medium

What is Clean Architecture, and is it overkill for Android?

Clean Arch separates layers (UI · Domain · Data) by responsibility, with dependencies pointing inward. It's overkill for small apps, valuable for large multi-module ones with shared business logic.

Three layers:

  • Data: data sources, DTOs, Repository implementations.
  • Domain: business rules, UseCases, entities. Pure Kotlin (no Android imports).
  • UI / Presentation: ViewModels, Composables.

Use Clean when: multi-module, KMP shared domain, large team, business logic complex enough to warrant explicit UseCases. Skip when: small CRUD apps where Repository is enough — UseCase becomes a 1-line wrapper around Repo.

Follow-up: "What's a UseCase?" A class with a single operator fun invoke(...) representing one business operation (GetUserByIdUseCase). Helps compose ViewModels from small testable pieces.
Q47Medium

What is Dependency Injection? Why use it?

DI = receiving your dependencies from outside instead of constructing them yourself. Makes code testable (swap real for fake), decoupled, and configurable.
// Without DI — hard to test, tight coupling
class UserRepository {
    private val api = Retrofit.Builder()...build().create()
}

// With DI — receive dependencies
class UserRepository(val api: UserApi) {
    // in tests, pass a fake
}

Manual DI works for small apps. Larger apps use a DI framework (Hilt, Koin) to wire dependencies automatically.

Q48Medium

Hilt vs Dagger vs Koin — when to pick what?

Hilt = Dagger preconfigured for Android (default in 2026). Dagger = more control, no Android opinions. Koin = pure Kotlin DSL (no compile-time validation).

Q49Hard

Hilt scopes: SingletonComponent, ViewModelComponent, ActivityRetainedComponent — what's the difference?

Each scope binds an object's lifetime to a specific component lifecycle. Pick the smallest scope that satisfies the dependency's needs.
  • SingletonComponent: lives for the entire app process. Use for OkHttpClient, Database, Retrofit.
  • ActivityRetainedComponent: survives configuration changes; cleared with the Activity. Use for shared state across Fragments hosted by the same Activity.
  • ViewModelComponent: scoped to a single ViewModel.
  • ActivityComponent: tied to Activity instance (re-created on rotation).
  • FragmentComponent: tied to Fragment instance.
  • ServiceComponent: tied to Service.
@Singleton @Provides
fun provideOkHttp(): OkHttpClient = OkHttpClient.Builder().build()

@ActivityRetainedScoped @Provides
fun provideSession() = SessionState()
Gotcha: Wrong scope = leaks. A @Singleton object holding an Activity reference will keep the Activity in memory forever.
Q50Hard

What is multibinding (@IntoSet, @IntoMap) in Hilt and what's it good for?

Multibinding lets multiple @Provides contribute to a single Set or Map. Used for plugin systems, interceptors, analytics handlers.
@Module @InstallIn(SingletonComponent::class)
object NetModule {
    @Provides @IntoSet
    fun auth(): Interceptor = AuthInterceptor()

    @Provides @IntoSet
    fun logging(): Interceptor = HttpLoggingInterceptor()

    @Provides @Singleton
    fun okHttp(interceptors: Set<Interceptor>): OkHttpClient =
        OkHttpClient.Builder().apply {
            interceptors.forEach { addInterceptor(it) }
        }.build()
}

You can add a new interceptor anywhere in the codebase by adding another @IntoSet binding — no changes needed to the OkHttpClient provider.

Follow-up: "When use @IntoMap?" When you need keyed selection — e.g., Map<Class, ViewModel.Factory> in pre-Hilt VM injection.
Q51Hard

Constructor injection vs field injection — which to prefer and why?

Constructor injection always. Field injection is only used when you can't control construction (Activities, Fragments — Android constructs them). Constructor wins on testability, immutability, and "you can't forget to inject".
// Constructor — preferred
class UserRepository @Inject constructor(
    private val api: UserApi,
    private val dao: UserDao
)

// Field — required for Android-instantiated classes
@AndroidEntryPoint
class MyActivity : ComponentActivity() {
    @Inject lateinit var analytics: Analytics
}

ViewModels with @HiltViewModel can use constructor injection because Hilt provides a Factory.

Gotcha: If you use field injection in a class you create yourself, the inject doesn't happen. Always pair with the right Hilt entry point.
Q52Hard

What's a multi-module Android project? Why split it?

Split your app into Gradle modules by feature (or by layer). Improves build time (parallel compilation, smaller incremental builds), enforces architecture (modules can't reach into each other's internals), and makes ownership clear.

Common module split: 

:app                       // thin glue — manifests, theme, NavHost
:core:network              // OkHttp, Retrofit setup
:core:database             // Room schema, DAOs
:core:design-system        // shared Composables, theme
:feature:home              // home screen + VM + repo
:feature:profile           // independent feature
:feature:settings
  • Build speed: changing one feature doesn't recompile others.
  • Encapsulation: features expose only public APIs; internal stays internal.
  • Dynamic feature modules for on-demand delivery.
Follow-up: "What's the trade-off?" More boilerplate (Gradle config, dependency wiring), and over-modularization can slow Gradle config phase.

6. Data & Persistence

Room, DataStore, migrations — the local-first stack.

Q53Easy

Room vs raw SQLite vs SharedPreferences vs DataStore — when to use which?

Room = SQL with type safety + Flow integration (default for structured data). DataStore = key/value or typed Proto, async (default for prefs). SharedPreferences = legacy, sync, deprecated for most uses. Raw SQLite = almost never.
Use
RoomAnything relational: lists of entities, joins, queries.
DataStore (Preferences)Simple K/V settings (theme, last login). Replaces SharedPreferences.
DataStore (Proto)Typed structured config (when you'd otherwise use a JSON file).

Q54Medium

How does Room work under the hood? What does the annotation processor generate?

Room is a compile-time SQLite wrapper. KSP/kapt reads your @Entity, @Dao, @Database classes and generates implementations: type converters, prepared statements, query result mappers, and a SupportSQLiteOpenHelper.
  • @Entity → table schema (CREATE TABLE).
  • @Dao with @Query("SELECT * FROM users") → generates a function that runs the prepared statement and maps rows to objects, validating SQL at compile time.
  • @Database → generates a class extending RoomDatabase that wires DAOs, manages migrations, and exposes the DB instance.
  • Flow/LiveData return types → Room observes the affected tables via an InvalidationTracker and re-emits when they change.
@Entity
data class User(@PrimaryKey val id: String, val name: String)

@Dao
interface UserDao {
    @Query("SELECT * FROM User WHERE id = :id")
    suspend fun get(id: String): User?

    @Query("SELECT * FROM User")
    fun observeAll(): Flow<List<User>>
}
Follow-up: "Why suspend functions?" Room runs queries on a background dispatcher (Dispatchers.IO); suspend ensures you don't block the main thread.
Q55Medium

Room migrations — how to add a column without losing data?

Bump the database version, write an explicit Migration object that runs ALTER TABLE statements, register it on the DatabaseBuilder. Test it with MigrationTestHelper.
val MIGRATION_1_2 = object : Migration(1, 2) {
    override fun migrate(db: SupportSQLiteDatabase) {
        db.execSQL("ALTER TABLE User ADD COLUMN email TEXT NOT NULL DEFAULT ''")
    }
}

Room.databaseBuilder(ctx, AppDatabase::class.java, "app.db")
    .addMigrations(MIGRATION_1_2)
    .build()
  • Export the schema (room.schemaLocation) — commit it to git for review and migration testing.
  • Never use fallbackToDestructiveMigration() in release builds — wipes user data.
  • For complex changes (table rename, column type change) write multi-step migrations.
Gotcha: Migration test gap is the #1 source of crashes for users on old versions. Always test the chain v1→v2→v3 with MigrationTestHelper.
Q56Medium

What are TypeConverters in Room?

Functions Room uses to read/write columns whose types it doesn't natively understand. Common cases: Date, List<String>, enums.
object Converters {
    @TypeConverter fun dateToLong(d: Date?): Long? = d?.time
    @TypeConverter fun longToDate(t: Long?): Date? = t?.let { Date(it) }
}

@Database(entities = [...], version = 1)
@TypeConverters(Converters::class)
abstract class AppDatabase : RoomDatabase() { ... }

Keep TypeConverters tiny and pure. If you find yourself doing JSON serialization in one, you're probably storing the wrong shape — consider a separate table or a Proto column.

Q57Hard

Room transactions — when and how?

Wrap multiple writes that must succeed together in a transaction. Use @Transaction on a Dao method or db.withTransaction { } for ad-hoc.
@Dao
interface OrderDao {
    @Insert suspend fun insertOrder(o: Order): Long
    @Insert suspend fun insertItems(items: List<Item>)

    @Transaction
    suspend fun insertOrderWithItems(o: Order, items: List<Item>) {
        val id = insertOrder(o)
        insertItems(items.map { it.copy(orderId = id) })
    }
}

// Or ad-hoc:
db.withTransaction {
    dao.deleteOld()
    dao.insertNew(items)
}

@Transaction is also required on multi-table queries with nested objects (returning List<OrderWithItems>), so reads see a consistent snapshot.

Follow-up: "What about concurrent writes from two coroutines?" Room serializes writes via a single writer thread; reads can be concurrent. No data races, but consider Mutex if you have read-modify-write logic.
Q58Hard

DataStore vs Proto DataStore — when use which?

Preferences DataStore = string-keyed, untyped (preferencesOf("theme" to "dark")). Proto DataStore = single typed object backed by a .proto schema, type-safe.
// Preferences — quick K/V
val THEME = stringPreferencesKey("theme")
context.dataStore.edit { it[THEME] = "dark" }
val theme: Flow<String?> = context.dataStore.data.map { it[THEME] }

// Proto — typed schema
// settings.proto:  message Settings { string theme = 1; bool notifs = 2; }
context.settingsStore.updateData { it.toBuilder().setTheme("dark").build() }
val settings: Flow<Settings> = context.settingsStore.data

Use Proto when you have a structured config object you'll evolve over time — schema versioning is much safer than scattered string keys.

Gotcha: DataStore is async (Flow). Don't try to read it synchronously in onCreate — block the main thread is wrong, defer to collect or first() in a coroutine.

7. Networking

OkHttp, Retrofit, interceptors, caching, WebSocket, certificate pinning.

Q59Easy

OkHttp vs Retrofit — what does each do?

OkHttp = the HTTP client (TCP, TLS, HTTP/2, connection pool, interceptors). Retrofit = a type-safe annotation-driven layer on top that turns interface methods into HTTP calls.
// Pure OkHttp
val req = Request.Builder().url("https://api.com/users/1").build()
val resp = okHttp.newCall(req).execute()

// Retrofit on top
interface UserApi {
    @GET("users/{id}")
    suspend fun get(@Path("id") id: Int): User
}
val api: UserApi = retrofit.create()
val u = api.get(1)

Retrofit handles JSON conversion (Gson/Moshi/Kotlinx serialization), suspend support, error handling helpers.

Q60Medium

OkHttp Interceptor vs Authenticator — what's the difference?

Interceptor = mutates every request/response (logging, headers, retry). Authenticator = called only when a 401 comes back; refresh the token and return the same request with a new auth header.
class AuthInterceptor(val tokens: TokenStore) : Interceptor {
    override fun intercept(chain: Interceptor.Chain): Response {
        val req = chain.request().newBuilder()
            .header("Authorization", "Bearer ${tokens.access()}")
            .build()
        return chain.proceed(req)
    }
}

class RefreshAuthenticator(val tokens: TokenStore) : Authenticator {
    override fun authenticate(route: Route?, response: Response): Request? {
        val newAccess = tokens.refresh() ?: return null   // give up
        return response.request.newBuilder()
            .header("Authorization", "Bearer $newAccess")
            .build()
    }
}

val client = OkHttpClient.Builder()
    .addInterceptor(AuthInterceptor(tokens))
    .authenticator(RefreshAuthenticator(tokens))
    .build()
Gotcha: Authenticator must be thread-safe — multiple parallel 401s can fire simultaneously. Single-flight the refresh with a Mutex.
Q61Medium

HTTP caching with OkHttp — Cache, ETag, Cache-Control

Give OkHttp a Cache directory; the server controls freshness via Cache-Control; OkHttp uses ETags for conditional GETs (304 Not Modified). For client-driven cache rules, add interceptors that rewrite headers.
val cache = Cache(File(ctx.cacheDir, "http"), 10L * 1024 * 1024)

val client = OkHttpClient.Builder()
    .cache(cache)
    .addNetworkInterceptor { chain ->
        val resp = chain.proceed(chain.request())
        resp.newBuilder()
            .header("Cache-Control", "public, max-age=300")
            .build()
    }

  • Use addNetworkInterceptor to rewrite response headers (e.g., force caching for endpoints whose server forgot Cache-Control).
  • Use addInterceptor to alter cache requests (e.g., force network refresh on pull-to-refresh).

Q66Hard

How do you implement request retry with exponential backoff in OkHttp?

Add an Interceptor that catches retryable failures (IOException, 5xx) and re-submits the request with growing delay + jitter, capped at N attempts.
class RetryInterceptor(val max: Int = 3) : Interceptor {
    override fun intercept(chain: Interceptor.Chain): Response {
        var attempt = 0
        while (true) {
            try {
                val resp = chain.proceed(chain.request())
                if (resp.code < 500 || ++attempt >= max) return resp
                resp.close()
            } catch (e: IOException) {
                if (++attempt >= max) throw e
            }
            val base = (1 shl attempt) * 200L
            Thread.sleep(base + Random.nextLong(0, base / 2))   // jitter
        }
    }
}

Don't retry on 4xx (won't change). Don't retry POST/PUT without an idempotency key (might double-write).

Gotcha: Retrying without jitter creates a thundering herd when many clients hit a flaky endpoint at once. Always add ±50% jitter.

How does Doze mode affect background work?

9. Performance, Memory & Battery

Where Senior diverges from Staff. Cold-start, leaks, frame drops, baseline profiles.

Q73Medium

What is a memory leak in Android? Most common causes?

An object outlives the scope it should belong to, holding references that prevent GC. Most common: an Activity is held by a static field, a long-lived listener, an inner class, or a Handler with delayed messages.
  • Static reference to Context: companion object { lateinit var ctx: Context }. Use applicationContext if needed.
  • Inner class holding outer: non-static inner classes implicitly hold a reference to the enclosing class. Use a static nested class or pass a WeakReference.
  • Handler with delayed runnables: pending messages keep the Handler alive, which keeps the Activity alive.
  • Coroutine that outlives the View: launching from a non-lifecycle scope.
  • Subscriptions to long-lived singletons (LocalBroadcastManager, EventBus) without unsubscribing.

Find them: LeakCanary (debug builds) — instant alerts on retained instances.

Gotcha: Activity context leaks are devastating because each retained Activity drags ~ MBs of view hierarchy.
Q74Medium

How does Android profiling work? What tools do you use?

Android Studio Profiler covers CPU, Memory, Network, Energy. Macrobenchmark for end-to-end metrics (cold start, frame timing). Perfetto for low-level traces. R8 mapping + symbolication for release builds.
  • CPU profiler: sample or instrument call stacks. Find slow methods and main-thread offenders.
  • Memory profiler: live allocation tracking; capture heap dumps to find leaks.
  • Macrobenchmark: write tests that measure cold/warm start, scrolling jank — runs on a real device against a release build.
  • Perfetto / Systrace: kernel + user-space trace. Best for understanding scheduler behavior.
  • StrictMode: catches main-thread disk/network access in debug.
Q75Medium

What's an ANR? How do you avoid them?

ANR = Application Not Responding. Triggered when the main thread is blocked > 5 seconds (input) or a Service/Receiver doesn't return in time. Avoid by never doing blocking work on the main thread.
  • Common causes: synchronous DB queries, network calls, large JSON parsing, image decoding, file I/O.
  • Fix: move work to a background dispatcher (withContext(Dispatchers.IO)).
  • StrictMode in debug catches main-thread offenses early.
  • Production ANRs are reported in Play Console / Crashlytics; treat them with the same urgency as crashes.
// Bad
override fun onCreate(b: Bundle?) {
    val users = db.userDao().getAllSync()
}

// Good
override fun onCreate(b: Bundle?) {
    lifecycleScope.launch { vm.load() }
}
Gotcha: Even fast disk reads can cause ANRs under heavy I/O contention. The rule is "no I/O on main, ever."
Q76Medium

How do you reduce app cold start time?

Minimize Application.onCreate work, defer SDK init via App Startup, use Baseline Profiles, lazy-init heavy singletons, avoid I/O in static blocks.
  • Application.onCreate: keep it under ~50 ms. Anything heavy → defer.
  • App Startup library: initialize ContentProviders cleanly without N forced eager init blocks.
  • Baseline Profiles: ship pre-compiled hot paths (AOT) — typical 20-40% cold start improvement.
  • Splash screen API (Android 12+): mandatory; replaces "blank screen" perception.
  • R8 / minification: strips dead code → smaller class loading time.
  • SplitInstall / dynamic feature modules: only download what's needed.
class App : Application() {

}

11. Library Internals (How They Actually Work)

This is where Senior screens get tested. Knowing why is the level signal.

Q87Hard

How does Glide work internally?

A pipeline: memory cache → disk cache → network → decode → transform → cache write → bind to target. Each stage short-circuits if it has the bitmap. Lifecycle-bound: requests are cancelled when the host fragment is destroyed.
  1. Request building: Glide.with(view).load(url).into(target) creates a Request object.
  2. Lifecycle attachment: an internal RequestManagerFragment hooks lifecycle to cancel requests on destroy.
  3. Cache lookup:
    • Active resources (in-use bitmaps held by WeakReference)
    • Memory cache (LRU, ~1/8 heap)
    • Resource disk cache (decoded + transformed)
    • Data disk cache (raw bytes)
  4. Network fetch via OkHttp/HttpURLConnection.
  5. Decode with downsampling (inSampleSize) targeting view dimensions.
  6. BitmapPool reuse: pre-allocate Bitmaps to slash GC churn.
  7. Transformation (resize, crop) → cache → set on target.

Cache key = URL + transformations + size + signature.

Q88Hard

How does OkHttp's connection pool work?

Connections are kept alive after a request and reused for subsequent requests to the same host. Default: max 5 idle, kept for 5 minutes. With HTTP/2, one connection multiplexes many concurrent requests.
  • OkHttp keeps a ConnectionPool per OkHttpClient.
  • For HTTP/1.1 each connection serves one request at a time; pool grows as needed.
  • For HTTP/2, one connection can serve many concurrent requests via streams — pool typically only holds one connection per host.
  • Connections in TIME_WAIT consume ports; reuse saves the TLS handshake (~100-200 ms saved per request on cellular).
  • Always use a single shared OkHttpClient across the app — creating new ones throws away the pool and the cache.
Gotcha: Never call okHttpClient.dispatcher.executorService.shutdown() in production — kills the pool for the rest of the app's life.
Q89Hard

How does Retrofit turn an interface into HTTP calls?

Retrofit uses dynamic proxy. retrofit.create(UserApi::class.java) returns a Proxy.newProxyInstance; method invocations are intercepted, the annotations parsed (lazily, then cached), and turned into HTTP calls.
  1. create() returns a Proxy implementing your interface.
  2. First call to a method triggers reflection: parse @GET/@POST annotations, parameters, return type adapter (suspend, Flow, Call).
  3. Built ServiceMethod is cached.
  4. Subsequent calls skip reflection; build an HTTP request from method args.
  5. Pass to OkHttp; convert response via the registered ConverterFactory (Moshi, Gson, kotlinx.serialization).

For suspend functions, Retrofit uses a CoroutineCallAdapterFactory (built-in) to bridge OkHttp's enqueue callback to suspendCoroutine.

Q90Hard

How does Compose runtime work? What's the slot table?

Compose's runtime maintains a slot table — a flat array of "slots" representing the UI tree. On recomposition, the compiler-generated code walks the slot table, comparing inputs; if equal, the slot is skipped (skipping recomposition).
  • Compose Compiler transforms @Composable functions, adding hidden parameters (Composer, key), and inserting calls to startRestartGroup/endRestartGroup around the body.
  • The Composer maintains the slot table and the position pointer.
  • remember stores into a slot; the value persists across recompositions because the slot is preserved.
  • State changes notify the Composer; only the smallest enclosing restartable scope is invalidated and re-run.
  • The slot table also serves positional memoization: two calls to the same Composable at the same position are matched by position+key.
Follow-up: "Why is key() needed in some loops?" Without an explicit key, Compose uses position; if items reorder, state attaches to the wrong items.
Q91Hard

How does Coroutines' suspend implementation actually work?

The compiler converts each suspend function into a state machine that takes a Continuation. Each suspension point becomes a labeled state; when the awaited operation completes, it calls resumeWith, which jumps to the next state.
// Source
suspend fun loadUser(): User {
    val u = api.fetch()           // suspends
    val profile = api.profile(u)  // suspends
    return u
}

// Roughly compiles to (simplified):
fun loadUser(cont: Continuation<User>): Any? {
    val sm = cont as StateMachine ?: StateMachine(cont)
    when (sm.label) {
        0 -> { sm.label = 1; val r = api.fetch(sm); if (r == COROUTINE_SUSPENDED) return r; sm.user = r as User }
        1 -> { sm.label = 2; val r = api.profile(sm.user, sm); if (r == COROUTINE_SUSPENDED) return r }
        2 -> return sm.user
    }
}

This is "Continuation-Passing Style" transformation — synchronous code in source, callback machinery at runtime.

Q92Hard

How does Room's Flow query react to data changes?

Room uses an InvalidationTracker that sets up SQLite triggers on the affected tables. When a write happens, the tracker fires; Flow subscribers re-execute the query and emit new results.
  • For each @Query("SELECT * FROM User") returning Flow, Room tracks the User table.
  • SQLite triggers fire on INSERT/UPDATE/DELETE → InvalidationTracker.
  • The Flow re-runs the query and emits new data on the IO dispatcher.
  • Multi-table queries (joins) track all involved tables.
  • Conflation: rapid writes don't cause N emissions; Room debounces.

This is why Room + Flow makes "single source of truth" so easy — UI just observes the table, never asks for refresh.

Gotcha: Room observes table-level changes, not row-level. A Flow from SELECT * FROM User WHERE id = 5 still re-emits when ANY user row changes.

val info = BiometricPrompt.PromptInfo.Builder()

What is the Play Integrity API?

Q100Hard

"Walk me through how you'd design a library like Glide / OkHttp from scratch."

Frame as: identify the pipeline, name the cache tiers, lifecycle binding, threading model, public API ergonomics, and extensibility points. The interviewer doesn't expect a complete impl — they want to see if you think in systems.

For an image library:

  1. Public API: fluent builder. ImageLoader.with(view).load(url).into(target).
  2. Request object: encapsulates URL, transformations, target size, priority, lifecycle owner.
  3. Cache tiers:
    • Active references (WeakReference of in-use bitmaps).
    • Memory LRU (~1/8 heap budget).
    • Disk LRU (decoded + transformed result).
    • Disk LRU (raw bytes for re-decode at different size).
  4. Cache key: URL + size + transformations + signature. Different sizes = different entries.
  5. Pipeline: try caches in order, then fetch (OkHttp), decode (with inSampleSize), transform, write back to caches, deliver to target.
  6. Threading: dispatcher thread for orchestration, IO pool for network/disk, decode pool for CPU.
  7. Lifecycle: hidden RetainedFragment per Activity to cancel in-flight requests on destroy. View tag tracks which request is current — late responses to a recycled view are dropped.
  8. Bitmap pool: reuse bitmap allocations to slash GC churn.
  9. Request coalescing: dedupe concurrent loads of the same key.
  10. Extensibility: pluggable Loaders (URL, Uri, File, Resource), Decoders, Transformations, Targets.

Then talk trade-offs: memory budget vs cache hit ratio, disk IO vs decode CPU, eager vs lazy evictions, etc.

For other library asks (OkHttp, Retrofit, Coroutines, Compose), follow the same pattern: public API → core abstractions → pipeline → caching/state → threading → lifecycle → extensibility → trade-offs.

Follow-up: "What would you skip for v1?" Disk caching, transformations beyond simple resize, GIF/SVG support, and prefetch. Ship a memory-cache-only proof first.

14. More Coroutines & Flow (Deep Dive)

The questions interviewers go to when they want to separate Senior from Staff.

Q101Medium

coroutineScope vs supervisorScope — what's the difference?

Both create a new scope and wait for all children. coroutineScope cancels all children if any one fails; supervisorScope isolates failures so siblings keep running.
// One failure cancels everything
coroutineScope {
    launch { riskyCallA() }   // if this throws...
    launch { riskyCallB() }   // ...this gets cancelled too
}

// Failures are isolated
supervisorScope {
    launch { riskyCallA() }   // if this throws, B keeps running
    launch { riskyCallB() }
}

Use coroutineScope for "all-or-nothing" parallel operations (parallel decomposition where any failure means nothing is valid). Use supervisorScope for "best-effort" parallel work (e.g., loading three optional widgets — failure of one shouldn't kill the others).

Follow-up: "What's the difference vs SupervisorJob()?" Same idea, but SupervisorJob is a Job you can pass to a CoroutineScope; supervisorScope is a suspending function that creates a temporary supervisor scope.
Q102Medium

withTimeout vs withTimeoutOrNull

Both cancel the inner block if it exceeds the time limit. withTimeout throws TimeoutCancellationException; withTimeoutOrNull returns null on timeout instead.
try {
    val user = withTimeout(5_000) { api.fetch() }
} catch (e: TimeoutCancellationException) {
    showError()
}

// Cleaner
val user = withTimeoutOrNull(5_000) { api.fetch() } ?: return showError()

The cancellation is cooperative — same rules as Q15. If your block doesn't suspend, the timeout won't fire until it does.

Gotcha: Don't catch the bare CancellationException superclass — you'll swallow normal cancellations. Catch TimeoutCancellationException specifically.
Q103Medium

runBlocking — when is it safe to use?

Tests, main() functions, and bridging suspend code into legacy synchronous APIs. Never in production app code on the main thread — it blocks until the block completes, which is what coroutines are designed to avoid.
// OK: in a unit test
@Test fun load() = runBlocking {
    val r = repo.fetch()
    assertEquals(...)
}

// OK: bridging from a synchronous Worker / library
override fun doWork(): Result = runBlocking {
    suspendingWork()
    Result.success()
}

// NOT OK: in your Activity to "make this synchronous"
val u = runBlocking { api.fetch() }    // blocks main, ANR risk

Use runTest (not runBlocking) for coroutine tests in 2026 — it controls virtual time for delays.

Q104Hard

Flow operators: buffer, conflate, retry, onEach — what each does

buffer = decouple producer/consumer with a queue. conflate = drop intermediate values when consumer is slow. retry = re-collect on failure (with optional predicate). onEach = side effect on each emission without changing the flow.
flow
    .onEach { Log.d("item", it.toString()) }    // log without consuming
    .buffer(capacity = 10)                       // async pipeline
    .conflate()                                  // drop old when slow consumer
    .retry(3) { e -> e is IOException }      // retry IO errors only
    .collect { render(it) }
  • buffer: producer doesn't wait for consumer; data sits in a queue. Useful when production and consumption rates differ.
  • conflate: when a new value arrives and the previous is unread, drop the previous. Perfect for "show latest only" UI state.
  • retry: re-collect from the upstream on failure. retryWhen for delays/backoff.
  • onEach: never mutates emissions; ideal for analytics/logging.
Follow-up: "When use collectLatest?" When each new emission should cancel the previous collector. Useful for "render the latest only" with cancellation of in-flight work.

15. Java, JVM & Concurrency Primitives

Even Kotlin-only shops still ask these — the JVM is what runs underneath.

Q105Easy

The equals + hashCode contract — why both?

If two objects are equals, they MUST have the same hashCode. Otherwise hash-based collections (HashMap, HashSet) break — the object can be put in but never found.
  • Reflexive: x.equals(x) = true.
  • Symmetric: x.equals(y) ⟺ y.equals(x).
  • Transitive: x→y, y→z ⟹ x→z.
  • Consistent: repeated calls return the same result (assuming objects don't change).
  • hashCode contract: equal objects must have equal hashCodes. Different objects may share a hashCode (hash collisions are fine).
// Kotlin data class — equals + hashCode auto-generated
data class User(val id: String, val name: String)

// Java — both must be overridden together
class User {
    @Override public boolean equals(Object o) { ... }
    @Override public int hashCode() { ... }    // IntelliJ → Generate
}
Gotcha: Putting an object into a HashSet, then mutating a field used in hashCode → contains returns false even though the object is in the set. Use immutable types as keys.
Q106Medium

Java & Kotlin interop — @JvmStatic, @JvmField, @JvmOverloads, @JvmName

Annotations that change how Kotlin code looks from Java. Without them, Java callers get wrappers, getters/setters, or extra parameters that feel awkward.
class Util {
    companion object {
        // Without @JvmStatic, Java sees: Util.Companion.create()
        @JvmStatic fun create() = Util()        // → Util.create()

        // Without @JvmField, Java sees: Util.Companion.getKEY()
        @JvmField val KEY = "x"                  // → Util.KEY
    }
}

// @JvmOverloads — generate overloads for default args
@JvmOverloads
fun greet(name: String, prefix: String = "Mr.") { ... }
// Java sees both greet(String, String) AND greet(String)

// @JvmName — rename for Java callers (avoid platform-name clash)
@JvmName("intList")
fun List<Int>.sum() = ...    // avoid clash with another String version
Follow-up: "What about @Throws?" Annotates a Kotlin function so Java callers see a checked exception. Without it, Java sees only unchecked exceptions and the compiler doesn't enforce try/catch.
Q107Medium

What does volatile do? When use it?

Marks a field so reads and writes go directly to main memory (not CPU caches), and establishes a happens-before relationship across threads. Use for simple "flag" variables shared between threads — NOT for compound operations (use AtomicInteger / Mutex).
// OK: simple flag
@Volatile var running: Boolean = true

// NOT OK: count++ is not atomic, even on a volatile
@Volatile var counter = 0
counter++                         // read + add + write = 3 ops, race

// Use atomic for compound ops
val counter = AtomicInteger(0)
counter.incrementAndGet()
Gotcha: Volatile gives visibility, not atomicity. Most "compare and update" patterns need AtomicReference or synchronized.
Q108Medium

synchronized, ReentrantLock, and Kotlin's Mutex — how to choose?

synchronized = JVM intrinsic, blocks the thread. ReentrantLock = more flexible (tryLock, fairness). Mutex = coroutine-aware (suspends instead of blocking).
// Java/Kotlin synchronized — blocks the thread
@Synchronized fun set(v: Int) { count = v }

// ReentrantLock — same semantics, more options
val lock = ReentrantLock()
lock.withLock { count = v }
if (lock.tryLock(1, TimeUnit.SECONDS)) { try { ... } finally { lock.unlock() } }

// Mutex — for coroutines (doesn't block the thread)
val mutex = Mutex()
mutex.withLock { count = v }     // suspends instead of blocking

In Coroutines code, always use Mutex. Blocking inside a coroutine wastes a thread that could run other coroutines.

Q109Medium

What's the Looper / Handler / MessageQueue trio?

A thread becomes "looper-driven" via Looper.prepare() / Looper.loop(). The MessageQueue holds Messages; the Handler posts to / pops from the queue. The main thread is set up this way at process start so the system can dispatch UI events to it.
// Post a runnable to the main thread
Handler(Looper.getMainLooper()).post { showToast() }

// Post with delay
mainHandler.postDelayed({ hideLoader() }, 2000)

// Make any thread looper-driven
val t = Thread {
    Looper.prepare()
    Handler(Looper.myLooper()!!) { msg -> ...; true }
    Looper.loop()
}
t.start()

In modern Android, prefer Coroutines + Dispatchers.Main. Direct Handler usage is mostly legacy.

Follow-up: "Why is Handler used in custom Views for animations?" Backwards compatibility — postOnAnimation hooks into the Choreographer's frame callbacks. Modern code uses Animator/Compose animations.
Q110Medium

HashMap vs ConcurrentHashMap vs SynchronizedMap

HashMap = not thread-safe (corruption + infinite loops on concurrent put). SynchronizedMap = whole-map lock per operation. ConcurrentHashMap = lock-striping, fine-grained — much higher concurrency.
val bad   = HashMap<K, V>()                  // race-prone
val sync  = Collections.synchronizedMap(HashMap())  // coarse
val conc  = ConcurrentHashMap<K, V>()         // good

// Atomic compound ops on ConcurrentHashMap
conc.compute("key") { _, v -> (v ?: 0) + 1 }
conc.putIfAbsent("key", value)
Gotcha: Iterating a SynchronizedMap requires you to manually synchronize on it — otherwise concurrent modification can throw or yield wrong results.
Q111Medium

Java checked vs unchecked exceptions — and Kotlin's stance

Java has checked exceptions (caller MUST handle or declare) and unchecked (RuntimeException + descendants). Kotlin treats all exceptions as unchecked — no throws declaration in signatures.
  • Java's checked exceptions were meant to enforce error handling, but in practice produced ugly try/catch/throws boilerplate and "swallow and re-throw as RuntimeException" anti-patterns.
  • Kotlin removed them; you handle exceptions where they make sense.
  • For Java interop, annotate Kotlin functions with @Throws(IOException::class) so Java callers see the checked exception.
  • Modern Kotlin idiom: prefer Result types or sealed Result classes over exceptions for "expected failures" (network, validation).
// Result type idiom
sealed interface ApiResult<out T> {
    data class Success<T>(val v: T) : ApiResult<T>
    data class Error(val e: Throwable) : ApiResult<Nothing>
}
Q112Hard

JVM Memory: Heap vs Stack, Generations, and what GC pauses mean for an Android app

Stack = per-thread, holds method frames + primitives. Heap = shared, holds objects, divided into Young (Eden + Survivor) and Old generations. Garbage collection pauses freeze your app — minimize allocations to minimize jank.
  • Most allocations die young (request-scoped). The Young generation is collected often (cheap "minor GC"); survivors get promoted to Old.
  • Old generation requires "major GC" — pauses of 50-200 ms on Android, dropping frames.
  • Modern ART uses concurrent copying GC — much shorter pauses than older Dalvik.
  • Allocation hotspots in Compose/RecyclerView scroll loops are the #1 jank cause. Use object pools (BitmapPool, ViewHolder), avoid lambdas in scroll callbacks, prefer primitives.

Find allocation hotspots: Android Studio Profiler → Memory → "Record allocations" while scrolling.

16. Design Patterns in Android

Where they appear in real Android code — not abstract theory.

Q113Medium

SOLID principles with Android examples

Five rules for code that's easy to change without breaking. Most Android architecture choices (Repository, Hilt, ViewModel) are SOLID applied.
  • S — Single Responsibility: a class has one reason to change. ViewModel manages state; Repository fetches data. Don't merge.
  • O — Open/Closed: open for extension, closed for modification. Compose modifiers, OkHttp interceptors — extend without editing the core.
  • L — Liskov Substitution: subtypes must be drop-in replacements. A FakeUserRepository must satisfy the same contract as the real one.
  • I — Interface Segregation: small focused interfaces over fat ones. Define UserRepository with just user methods, not "AppRepository" with everything.
  • D — Dependency Inversion: depend on abstractions, not concretions. ViewModel depends on the UserRepository interface, Hilt picks the implementation.
Q114Easy

Singleton in Android: object vs Hilt @Singleton vs static — which to use?

For pure stateless utilities use Kotlin object. For singletons that have dependencies, use @Singleton in Hilt — gives you injectable dependencies + testability + scope clarity.
// Kotlin object — static-like, no DI needed
object StringUtils {
    fun isNullOrBlank(s: String?) = s.isNullOrBlank()
}

// Hilt singleton — has dependencies, testable
@Singleton
class UserRepository @Inject constructor(val api: Api, val dao: Dao)

Avoid manual hand-rolled singletons (private constructor + getInstance()) — testing nightmare, no DI integration, easy to leak Context.

17. Views, RecyclerView & Custom Drawing

Compose dominates new code, but most large apps still have View-based screens. Senior interviewers will ask.

Q120Medium

The View lifecycle: onMeasure, onLayout, onDraw

Three phases per frame: measure (how big do I want to be?), layout (where do my children go?), draw (paint pixels). All on the main thread. Each must be fast (sub-ms).
  • onMeasure(widthSpec, heightSpec): parent passes constraints (EXACTLY / AT_MOST / UNSPECIFIED). View calls setMeasuredDimension.
  • onLayout(changed, l, t, r, b): view positions itself and (if ViewGroup) calls layout on each child.
  • onDraw(canvas): view paints itself onto the supplied Canvas.
  • requestLayout(): re-runs measure + layout pass for the affected subtree.
  • invalidate(): re-runs draw only.
override fun onMeasure(wSpec: Int, hSpec: Int) {
    val w = MeasureSpec.getSize(wSpec)
    val h = MeasureSpec.getSize(hSpec).coerceAtLeast(200)
    setMeasuredDimension(w, h)
}
Gotcha: Allocating in onDraw (new Paint, new Path) destroys frame rate. Allocate in init/onLayout, mutate in onDraw.
Q121Medium

Touch event dispatch: dispatchTouchEvent → onInterceptTouchEvent → onTouchEvent

Touch starts at the Activity, descends through ViewGroups. Each ViewGroup can intercept (steal the event from children) or pass it down. The deepest View handling the event "consumes" it; ACTION_UP returns to the consumer.
Activity.dispatchTouchEvent
   └─ Window.superDispatchTouchEvent
        └─ ViewGroup.dispatchTouchEvent
             ├─ onInterceptTouchEvent(ev)            // steal? false → pass to child
             ├─ child.dispatchTouchEvent(ev)         // recurse
             └─ onTouchEvent(ev)                     // fallback if no child consumed

Use requestDisallowInterceptTouchEvent(true) to tell parents not to steal events mid-gesture (e.g., a horizontal swipe inside a vertically scrolling parent).

Follow-up: "How do you implement a SwipeRefreshLayout?" Override onInterceptTouchEvent to detect a downward drag at the top of the list, then steal future events to drag the refresh indicator.
Q122Medium

What's a Custom View? When write one?

Subclass View (or a built-in subclass) when you need custom drawing/measurement that built-in views can't compose. Common: charts, gauges, custom progress indicators, complex tappable diagrams.
class CircleView(ctx: Context, attrs: AttributeSet?) : View(ctx, attrs) {
    private val paint = Paint().apply {
        color = Color.RED
        isAntiAlias = true
    }
    override fun onDraw(c: Canvas) {
        val r = (width.coerceAtMost(height) / 2).toFloat()
        c.drawCircle(width / 2f, height / 2f, r, paint)
    }
}
  • Use obtainStyledAttributes to read XML attributes (requires attrs.xml declaration).
  • Implement onSaveInstanceState/onRestoreInstanceState to survive rotation.
  • For 2026 greenfield work, prefer Compose — far less ceremony for the same outcome.
Q123Medium

RecyclerView vs ListView — and how RecyclerView reuses views

RecyclerView uses ViewHolder pattern + a recycle pool. As items scroll off-screen, their ViewHolders go back to the pool; new visible items get bound to recycled VHs (no inflation). Result: scroll = bind only, ~10× faster than ListView.
  • onCreateViewHolder: inflates a new VH (called only ~visible-count + 2 times).
  • onBindViewHolder(vh, position): re-binds a recycled VH to new data.
  • RecycledViewPool: shared across RecyclerViews (e.g., nested horizontal lists in a vertical feed).
  • setHasFixedSize(true): hint that adapter changes don't change RV size — skips remeasure.
  • setItemViewCacheSize: controls how many off-screen VHs are kept ready.
Gotcha: Always setHasStableIds(true) + provide stable item IDs for animations to work right after data changes.
Q124Medium

DiffUtil — what does it solve?

Compute the minimum set of changes (insert/remove/move/change) between an old and a new list, then dispatch them to RecyclerView for efficient animations and to avoid notifyDataSetChanged().
class UserDiff : DiffUtil.ItemCallback<User>() {
    override fun areItemsTheSame(o: User, n: User) = o.id == n.id
    override fun areContentsTheSame(o: User, n: User) = o == n
}

val adapter = ListAdapter<User, UserVH>(UserDiff()) { ... }
adapter.submitList(newList)         // diff computed in background

ListAdapter wraps DiffUtil + AsyncListDiffer; computation happens on a background thread, dispatch on main. Without DiffUtil, every list update calls notifyDataSetChanged, which kills animations.

Follow-up: "What's areContentsTheSame returning false but areItemsTheSame true?" Same item, different content → triggers a "change" event that you can use for partial binding via getChangePayload.
Q125Medium

ConstraintLayout vs LinearLayout vs RelativeLayout — when use what?

ConstraintLayout for almost everything (flat hierarchy, performant). LinearLayout for simple stacks. RelativeLayout = legacy, prefer ConstraintLayout. FrameLayout for overlapping layers.
  • ConstraintLayout: define positions via constraints to other views or parent. Flat hierarchy = faster measure/layout.
  • LinearLayout: simple linear stacks. Avoid layout_weight in nested hierarchies — double measure pass.
  • FrameLayout: stacking (overlay loading spinner over content).
  • CoordinatorLayout: AppBar/scroll behaviors (collapsing toolbar, FAB).

With Compose, layout is even flatter — Box, Row, Column, ConstraintLayout (yes, ported).

Q126Medium

findViewById vs ViewBinding vs DataBinding

findViewById = unsafe (NPE if id missing). ViewBinding = compile-time safe, lightweight. DataBinding = ViewBinding + 2-way bindings + expression language; heavier and Compose has largely replaced its niche.
// findViewById — fragile
val tv: TextView = findViewById(R.id.title)

// ViewBinding — auto-generated, safe
val binding = ActivityMainBinding.inflate(layoutInflater)
binding.title.text = "Hello"

// DataBinding — XML expressions and 2-way binding

Q127Hard

Intents, BroadcastReceiver, ContentProvider, Notifications, Permissions — the Android framework you live in.

Q128Easy

Explicit vs Implicit Intent — when use which?

Explicit = name the target Activity by class name (in-app navigation). Implicit = describe what you want (action + data); the system finds an app that matches.

    putExtra("userId", "42")
}

// Implicit — let the system pick (chooser, default app, etc.)

    type = "text/plain"
    putExtra(Intent.EXTRA_TEXT, "Check this out")

Q129Medium

Deep Links vs App Links vs Universal Links

Q130Medium

Q133Medium

Runtime Permissions — request flow + 2026 specifics (POST_NOTIFICATIONS, partial photos, full-screen intent)

Q134Medium

What's ActivityResultContracts and why use it instead of onActivityResult?

Modern, type-safe replacement for the deprecated startActivityForResult + onActivityResult pattern. Each contract is a typed launcher; results survive process death.
// Old (deprecated)
startActivityForResult(intent, REQUEST_CODE)
override fun onActivityResult(req: Int, res: Int, data: Intent?) { ... }

// New

    ActivityResultContracts.GetContent()

Built-in contracts: GetContent, PickContact, RequestPermission, RequestMultiplePermissions, StartActivityForResult, TakePicture, etc. You can also write a custom contract.

Q135Hard

Service IPC: Messenger vs AIDL vs Bound Service

Bound Service = client/server within same process (Binder, no marshalling). Messenger = simple cross-process communication (queued, single-threaded). AIDL = full-blown cross-process API with parallel calls; required when you need multi-threading on the server.
  • Bound Service: bindService + onBind() returning a custom Binder. Same process only.
  • Messenger: handles cross-process via a single Handler. Simpler but serial.
  • AIDL: define an .aidl interface; framework generates stubs + proxies. Use when calls need to run concurrently on the server.

Most apps don't write AIDL. It's needed for system-level integrations (SDKs that run a service, system apps).

Follow-up: "What's a Binder?" Android's IPC primitive at the kernel level. Single shared memory + driver routing. The basis of all cross-process communication on Android.

Q138Medium

APK vs AAB (Android App Bundle) — which to ship?

Mandatory: AAB on the Play Store since 2021. Google generates per-device-config APKs from your bundle (only the right density, ABI, language). Result: ~25% smaller install size on average.
  • AAB: a publishing format (.aab). Contains all code + resources, but split per config.
  • Per-device APKs generated by Play, not by you.
  • Dynamic Feature Modules: only download a feature when needed (e.g., AR view).
  • Play App Signing (mandatory for new apps): Play holds the signing key; you upload signed with your upload key.

For internal distribution (Firebase App Distribution, internal QA), generate APKs from the AAB via bundletool.

Q139Medium

KAPT vs KSP — which annotation processing should you use?

KSP. It's 2-3× faster than KAPT, has better incremental support, and is the official direction for Kotlin annotation processing. Most major libraries (Hilt, Room, Moshi) now support KSP.
plugins { id("com.google.devtools.ksp") version "2.0.0-1.0.21" }

dependencies {
    ksp("androidx.room:room-compiler:2.6.1")
    ksp("com.google.dagger:hilt-compiler:2.51")
}
  • KAPT runs as a Java annotation processor — Kotlin code is converted to Java stubs first (slow).
  • KSP works directly on Kotlin sources — much faster, more accurate type info.
  • Library must be ported to KSP (most are now).
Q140Hard

What's R8's -keep rule for and when do you need it?

Tells R8 not to remove or rename classes/members it can't see being used (typically reflection-based code). Without proper -keep rules, R8 strips the symbol and crash at runtime.
// Keep all classes in a package + their members
-keep class com.example.dto.** { *; }

// Keep only the class names (not members) — for class-name reflection
-keepnames class com.example.events.*

// Keep based on annotation
-keep @com.example.SerializeMe class *

// Keep enum values (commonly stripped accidentally)
-keepclassmembers enum * {
    public static **[] values();
    public static ** valueOf(java.lang.String);
}
  • Most modern libraries (Moshi-codegen, kotlinx.serialization, Retrofit) work without manual rules — they ship their own consumer-rules.pro.
  • Reflection on your own DTOs always needs -keep.
  • Test release builds! It's the only way to catch missing rules before users do.

20. Compose Deep Dive

The deeper Compose questions that separate "I use Compose" from "I understand Compose".

Q141Hard

The three Compose phases: Composition, Layout, Drawing

Composition (run @Composable functions, build the tree). Layout (measure + place). Drawing (paint). State reads tracked at each phase — reading a value during Layout-only doesn't trigger recomposition.
  • Composition: @Composable functions execute, slot table updated. Triggered by state read inside a composable.
  • Layout: Measure + Place. Triggered by state read during measurement (e.g., reading scrollState.value in modifier.offset).
  • Drawing: Final paint to the canvas. Cheapest phase to invalidate.
// Bad: scroll position read in composition → recompose every frame
Text("Offset: ${scrollState.value}")

// Good: read in drawing phase → just redraw, no recomposition
Box(Modifier.offset { IntOffset(0, scrollState.value) })

Defer state reads to the lowest phase needed. Lambdas inside Modifier.offset { } and graphicsLayer { } run in later phases.

Q142Hard

What is graphicsLayer and when use it?

A modifier that puts a Composable on its own GPU layer — transformations (rotation, scale, alpha) happen on the GPU without recomposition. Essential for smooth animations.
Box(
    Modifier.graphicsLayer {
        rotationZ = animatedAngle
        alpha = animatedAlpha
        translationY = scrollOffset.toFloat()
    }
)
  • Reads inside the lambda are deferred to the drawing phase — no composition, no layout, just GPU transforms.
  • Animating Modifier.rotate(angle) directly causes layout invalidations; using graphicsLayer { rotationZ = angle } doesn't.
  • Compose's animation APIs (animateFloatAsState) often pair with graphicsLayer for performance.
Q143Medium

Compose theming: MaterialTheme, Material3, dynamic color

MaterialTheme provides Color, Typography, Shapes via CompositionLocal. Material3 (the modern set) supports Dynamic Color on Android 12+ — themes generated from the user's wallpaper.
@Composable
fun AppTheme(content: @Composable () -> Unit) {
    val dark = isSystemInDarkTheme()
    val colors = when {
        Build.VERSION.SDK_INT >= 31 ->
            if (dark) dynamicDarkColorScheme(LocalContext.current)
            else dynamicLightColorScheme(LocalContext.current)
        dark -> darkColorScheme()
        else -> lightColorScheme()
    }
    MaterialTheme(colorScheme = colors, typography = AppTypography, content = content)
}

Custom design tokens beyond Material? Use a custom CompositionLocal or extend MaterialTheme.

Q144Medium

Compose animations: animate*AsState, updateTransition, AnimatedVisibility, AnimatedContent

animate*AsState = single-value animation. updateTransition = coordinate multiple values. AnimatedVisibility = enter/exit a Composable. AnimatedContent = swap between Composables with cross-fade-style transitions.
// Single value — most common
val alpha by animateFloatAsState(if (visible) 1f else 0f)

// Coordinated multi-value
val transition = updateTransition(state, label = "main")
val color    by transition.animateColor { if (it == State.A) Red else Blue }
val radius   by transition.animateDp    { if (it == State.A) 12.dp else 36.dp }

// Show/hide
AnimatedVisibility(visible) { Card(...) }

// Swap content with transition
AnimatedContent(targetState = page) { p -> PageContent(p) }
Gotcha: Don't animate height to "wrap content" — use AnimatedVisibility or animateContentSize().
Q145Hard

Interop: AndroidView in Compose and ComposeView in XML

Use AndroidView to host a View inside Compose (MapView, AdView, custom legacy View). Use ComposeView in XML to drop a Composable into a View hierarchy. Bridges your migration both ways.

    factory = { ctx -> MapView(ctx).apply { onCreate(null); getMapAsync { ... } } },

// Host a Composable in XML

    android:id="@+id/compose_root"
    android:layout_width="match_parent"
    android:layout_height="wrap_content" />

  • AndroidView is heavier than pure Compose; reach for it only when needed.
  • Pass state via update lambda — runs whenever a captured State changes.

22. Classic GoF Design Patterns — Where in Android & Real Libraries

These come up at SDE-2/SDE-3 with the framing: "What is X pattern? Where do you see it in Android or in a library you use?" Knowing the library example is the difference between "I read about it" and "I use it."

Q151Medium

Proxy Pattern — and how Retrofit uses it

A Proxy stands in for a real object, intercepting calls so it can do something extra (lazy creation, security, remote calls, lazy loading). Retrofit's create() uses Java's Proxy.newProxyInstance to generate an interface implementation at runtime — every method call becomes an HTTP request.
interface UserApi {
    @GET("users/{id}")
    suspend fun get(@Path("id") id: Int): User
}

val api: UserApi = retrofit.create(UserApi::class.java)
// You never wrote `class UserApiImpl : UserApi`. Retrofit creates a Proxy.
// Every call to api.get(1) is intercepted by an InvocationHandler that:
//   1. Reads the method's annotations (@GET, @Path)
//   2. Builds a Request from the args
//   3. Hands it to OkHttp
//   4. Converts the response via Moshi/Gson

val u = api.get(1)        // looks like a method call, is actually HTTP

Other Android examples: Glide's RequestManager proxies the actual loading work tied to a lifecycle owner. Mockito's mocks are proxies that record method calls.

Follow-up: "Why prefer Proxy over a generated interface implementation at compile time?" Retrofit-style proxy keeps the API surface tiny (just an interface). The compile-time alternative is annotation processing (KSP) — what Hilt does. Trade-off: runtime flexibility vs build-time performance + safety.
Gotcha: Dynamic proxies use reflection — that's why Retrofit needs -keep ProGuard rules for your API interfaces.
Q152Medium

Chain of Responsibility — and OkHttp Interceptors

A request travels through a chain of handlers; each can act on it (mutate, short-circuit, log) and then pass it to the next. OkHttp Interceptors are the textbook example: every request walks application interceptors → core → network interceptors → network, then unwinds back through them.
class AuthInterceptor : Interceptor {
    override fun intercept(chain: Interceptor.Chain): Response {
        val req = chain.request().newBuilder()
            .header("Authorization", "Bearer $token")     // pre-process
            .build()
        val resp = chain.proceed(req)                         // pass to next
        // post-process here if needed
        return resp
    }
}

val client = OkHttpClient.Builder()
    .addInterceptor(LoggingInterceptor())   // runs first
    .addInterceptor(AuthInterceptor())
    .addInterceptor(RetryInterceptor())     // runs last (closest to network)
    .build()

Other examples: Android's touch event dispatch (Q121: dispatchTouchEvent → onInterceptTouchEvent → onTouchEvent up the chain). Compose's Modifier chain. Servlet filters in Java backends.

Follow-up: "What's the difference between application interceptors and network interceptors?" App interceptors run once per call (don't see redirects/retries). Network interceptors run for every actual network request (see redirects, retries, can observe 304 cache hits).
Q153Medium

Template Method — and where Android uses it

A base class defines the skeleton of an algorithm with hook methods that subclasses override. The shape of the flow is fixed; the steps are pluggable. Activity / Fragment lifecycle hooks are pure Template Method.
// You override hooks; the framework calls them in the right order.
class MyActivity : ComponentActivity() {
    override fun onCreate(b: Bundle?) { super.onCreate(b); /* hook */ }
    override fun onStart()   { super.onStart() }
    override fun onResume()  { super.onResume() }
    override fun onPause()   { super.onPause() }
    override fun onStop()    { super.onStop() }
}

// RecyclerView.Adapter — same pattern
class UserAdapter : RecyclerView.Adapter<UserVH>() {
    override fun onCreateViewHolder(...): UserVH = ...
    override fun onBindViewHolder(vh: UserVH, pos: Int) = ...
    override fun getItemCount(): Int = ...
}

Other examples: AsyncTask's doInBackground / onPostExecute (legacy). JobIntentService. Espresso's ActivityScenarioRule. Compose Worker's doWork().

Follow-up: "Why is Template Method common in Android?" Because the framework owns the lifecycle and the threading; you only get to plug in behavior at well-defined points.
Q154Medium

Composite Pattern — and the View hierarchy / Compose tree

Treat individual objects (leaves) and groups of objects (composites) uniformly through a common interface. Android's View / ViewGroup hierarchy is textbook: a ViewGroup IS a View and contains Views. You can call setVisibility(), requestLayout() on either.
// Same interface for leaf (View) and composite (ViewGroup)
val root: View = findViewById(R.id.root)
root.setVisibility(View.GONE)          // works whether root is a Button or a LinearLayout

// ViewGroup just adds collection-of-children semantics
class CustomLayout(ctx: Context) : ViewGroup(ctx) {
    override fun onLayout(...) {
        for (i in 0 until childCount) {
            getChildAt(i).layout(0, top, right, bottom)
        }
    }
}

Other examples: Compose tree (Box contains other Composables). View tree of HTML / SwiftUI. JSON tree (object/array vs primitive). Filesystem (folder/file).

Follow-up: "Trade-off?" Uniform interface is convenient but can leak collection-only methods (e.g., addView) onto leaves where they don't make sense. Compose handles this better — children are passed as a content lambda rather than added imperatively.
Q155Hard

Flyweight Pattern — RecyclerView pool, BitmapPool, Drawable.ConstantState

Share fine-grained objects across many contexts to save memory. Don't create a new heavy object per use — pool/share them. RecyclerView's recycled view pool, Glide's BitmapPool, and Drawable.ConstantState are all flyweights.
// Drawable.ConstantState — shared state across multiple Drawable instances
val d1 = ContextCompat.getDrawable(ctx, R.drawable.icon)!!
val d2 = ContextCompat.getDrawable(ctx, R.drawable.icon)!!
// d1 and d2 are different instances, but share the underlying bitmap data
// Mutating one (setTint) affects the other UNLESS you call mutate() first
d1.mutate().setTint(Color.RED)

// RecyclerView.RecycledViewPool — share ViewHolders across nested RVs
val sharedPool = RecyclerView.RecycledViewPool()
nestedRv1.setRecycledViewPool(sharedPool)
nestedRv2.setRecycledViewPool(sharedPool)

Glide's BitmapPool reuses Bitmap allocations across requests — without it, scrolling an image-heavy feed would create thousands of bitmaps and overwhelm the GC. String interning on the JVM is also a Flyweight.

Gotcha: Forget to call Drawable.mutate() and styling one drawable instance changes every other instance loaded from the same resource — common bug source.
Q156Medium

State Pattern — and modeling UI state with sealed classes

An object's behavior changes based on internal state, with each state encapsulated in its own type. The class delegates to the current state. Modern Android does this with sealed classes / sealed interfaces representing UI states; the UI when branches on the state type.
sealed interface UiState {
    data object Loading : UiState
    data class Success(val users: List<User>) : UiState
    data class Error(val msg: String) : UiState
}

@Composable fun Screen(state: UiState) {
    when (state) {                  // exhaustive
        UiState.Loading    -> Spinner()
        is UiState.Success -> UserList(state.users)
        is UiState.Error   -> ErrorView(state.msg)
    }
}

Other Android examples: ATM / Vending Machine state machines (system design Q33, Q34). MediaSession's PlaybackState. ConnectionState in WebSocket libraries. Compose's Lifecycle.State enum.

Follow-up: "Sealed class vs enum for state?" Enum if all states have the same shape (just labels). Sealed when each state carries different data — Loading has none, Success has a list, Error has a message.

Command Pattern — Runnable, Intent, WorkRequest

The Compose Snapshot system — how does Compose actually know which State you read?

Every State<T> is a "snapshot object". Reads inside a snapshot scope are tracked via thread-local SnapshotObserver. Each RecomposeScope records its read set; when a tracked State changes, only scopes that read it get invalidated. Writes also go through snapshots — so concurrent reads see a consistent view (like MVCC).
  • Snapshot = a versioned view of all state at a point in time.
  • Read tracking: when you call state.value inside a Composable, Compose's current SnapshotObserver records (state, currentRecomposeScope) in a map.
  • Write tracking: setting state.value = x creates a new "modified" snapshot record. After commit, observers fire for affected reads.
  • Invalidation: only the smallest enclosing RecomposeScope that read the state is marked dirty — that's why two Texts in a Column can re-run independently.
  • Snapshot.takeMutableSnapshot: lets you mutate state in isolation, then atomically apply (or abandon) — used internally by Compose for animations and concurrent recomposition.
// Concrete: this only re-runs the inner Text, not Counter
@Composable fun Counter() {
    val a = remember { mutableStateOf(0) }
    val b = remember { mutableStateOf(0) }
    Column {
        Text("A: ${a.value}")         // reads a → only this scope tracks a
        Text("B: ${b.value}")         // reads b → only this scope tracks b
    }
}
// a.value++ invalidates only the first Text; b.value++ only the second.
Follow-up: "Why are derivedStateOf reads tracked differently?" derivedStateOf creates a snapshot-aware computed value — its dependencies are tracked, and it only emits when the result actually changes. Without it, every dependency change re-runs the reader.

What does the Compose Compiler actually transform?

Three transforms on every @Composable function: (1) inject hidden Composer + $changed parameters, (2) wrap body in startRestartGroup/endRestartGroup + updateScope for invalidation, (3) generate skipping bytecode that compares parameters bit-by-bit and skips re-execution if nothing relevant changed.
// You write:
@Composable fun Greeting(name: String) {
    Text("Hello $name")
}

// Compiler emits roughly:
fun Greeting(name: String, $composer: Composer, $changed: Int) {
    $composer.startRestartGroup(0xABCD)
    val dirty = $changed
    if ($changed and 0b1 == 0)
        dirty = dirty or if ($composer.changed(name)) 0b100 else 0b10

    if (dirty and 0b11 != 0b10 || !$composer.skipping) {
        Text("Hello $name", $composer, 0)
    } else {
        $composer.skipToGroupEnd()
    }
    $composer.endRestartGroup()?.updateScope { c, _ ->
        Greeting(name, c, $changed or 1)
    }
}
  • $changed bitmask: each parameter gets 3 bits — "changed", "static", "force re-compute".
  • updateScope lambda: how Compose re-invokes only this function on invalidation, with the same args.
  • skipping: if all params are "stable" and unchanged, skip the entire body.
Follow-up: "What's strong skipping (Compose 1.6+)?" Even unstable params are now considered "skipable" if their equals() returns true. Reduces the need for stability annotations significantly.

ART: JIT vs AOT vs Profile-Guided Compilation

ART (Android Runtime) starts interpreted. Hot methods get JIT-compiled at runtime. Profiles of "what's hot" are recorded and uploaded to Play; on next install/idle, ART AOT-compiles those methods to native ahead of time. Baseline Profiles let you ship a curated profile in the APK so day-1 users get AOT performance.
  1. First launch: code runs interpreted (slow but starts instantly).
  2. Hot methods: JIT compiles them after some calls; subsequent calls run native.
  3. Profile recording: ART writes profile.prof per app — which methods were JIT'd, which classes loaded.
  4. Idle compilation: when device is charging + idle, ART AOT-compiles methods in the profile (and other hot ones).
  5. Profile cloud: Play aggregates profiles across users, distributes them via Play Cloud Profile.
  6. Baseline Profile: a profile YOU ship in the APK (baseline-prof.txt) — pre-compiled at install time.

Net effect of Baseline Profile: 20-40% cold start improvement, 30-50% scroll jank reduction on key flows. Compose-heavy apps benefit even more.

Follow-up: "Why does Compose particularly benefit?" Compose runtime is small but called millions of times. Pre-AOT'ing it means no JIT-warmup tax during the first key frames.

Java Memory Model: happens-before, volatile semantics, instruction reordering

JVM reorders reads/writes for performance. The JMM defines happens-before relationships that programmers can rely on across threads. volatile establishes happens-before across reads/writes. synchronized establishes it across lock release/acquire.
  • Without happens-before: thread A writes x then y; thread B reading y then x might see x as unset (compiler reordered, or B's CPU cache).
  • volatile: a read sees the latest write; AND all writes done before the volatile write are visible to anyone reading after.
  • synchronized: monitor exit happens-before next monitor enter on same lock.
  • final: writes in constructor happen-before any thread sees the constructed object.
  • Atomic*: use lock-free CAS; provide same memory ordering as volatile.
// Classic broken double-checked locking (pre-Java 5)
class Singleton {
    companion object {
        private var instance: Singleton? = null
        fun get(): Singleton {
            if (instance == null) {
                synchronized(Singleton::class.java) {
                    if (instance == null) instance = Singleton()
                }
            }
            return instance!!     // race: another thread might see partly-constructed instance
        }
    }
}

// Fixed: @Volatile (acts as a memory barrier)
@Volatile private var instance: Singleton? = null

// Better: just use Kotlin object — JVM class init guarantees thread-safety
object Singleton

Lock-free programming: CAS, AtomicReference, LongAdder vs AtomicLong

Compare-And-Swap (CAS) is a CPU primitive: "atomically set X to new IF X is currently old." Atomic classes use it to update without locks — multiple threads retry until they win. LongAdder shards the counter across CPU cores to slash contention vs AtomicLong.
// AtomicReference — basic CAS
val ref = AtomicReference<State>(initial)
fun update(transform: (State) -> State) {
    while (true) {
        val old = ref.get()
        val new = transform(old)
        if (ref.compareAndSet(old, new)) return      // won race
        // lost race — loop and retry with the latest value
    }
}

// AtomicLong — counter shared across threads, contention point
val count = AtomicLong()
count.incrementAndGet()                                     // CAS loop under contention

// LongAdder — sharded, per-thread cell + sum on read. Way faster under contention.
val count = LongAdder()
count.increment()                                            // hits its own cell
count.sum()                                                  // only when reading
  • AtomicLong is fine for low contention. Under high contention (counter pinged from 100s of threads), LongAdder is 10-100× faster.
  • Trade-off: LongAdder's sum() is O(cells), AtomicLong's get() is O(1).
  • Coroutines code rarely needs these directly — Mutex and StateFlow handle most cases.

Android's Concurrent Copying GC — and how to write GC-friendly code

ART uses Concurrent Copying GC since Android 8 — collects the young generation concurrently with the app, with very short stop-the-world pauses (typically < 5 ms). To stay friendly: minimize allocations in hot paths (scroll, animation, recomposition).
  • Young generation: most allocations die young; collected often, cheaply.
  • Old generation: long-lived objects; collected rarely but more expensively.
  • Concurrent: collector runs alongside the app threads, not in a hard stop-the-world.
  • Copying: live objects copied to a new region; old region freed in bulk. Avoids fragmentation.
  • Pauses: ~1-5 ms for typical young collections; the full pause budget you have at 60 fps is 16.6 ms — leaves little slack.

How to be GC-friendly:

  • Avoid allocations in tight loops (scroll handlers, onDraw, Compose recomposition).
  • Reuse objects: BitmapPool, ViewHolder pool, StringBuilder reuse.
  • Prefer primitives over boxed types in hot paths (Int over Integer, IntArray over List<Int>).
  • Don't capture in lambdas inside scroll handlers — every frame allocates a new Function instance.
  • Use SparseArray<V>, ArraySet, ArrayMap for small collections instead of HashMap (less allocation).
Gotcha: Allocation profiler (Android Studio) lies on debug builds — Compose generates many extra allocations in debug for state inspection. Always measure release.

App startup deep dive: Zygote, process fork, and what happens before onCreate

Android boots one Zygote process at boot that pre-loads core classes + framework. When you tap an app, system fork()s Zygote (almost instant — copy-on-write), the new process becomes your app, ContentProviders init, then Application.onCreate, then your launch Activity's onCreate.
  1. Zygote pre-loads framework classes, theme, common resources at boot.
  2. fork(): kernel copies the Zygote's address space (CoW); the child becomes your app process. Very fast — milliseconds.
  3. ART process init: relinks JNI, loads your APK's classloader.
  4. ContentProviders init: all ContentProviders declared in your manifest run their onCreate BEFORE Application.onCreate. (App Startup library uses this — it's the only ContentProvider you need.)
  5. Application.onCreate.
  6. Activity.onCreate: instantiate Activity, restore SavedInstanceState, run user code.
  7. First frame: layout + draw, then "time to first frame" stops.

Cold start = process forked + everything above. Warm start = process alive but Activity destroyed. Hot start = Activity alive, just brought to foreground.

Follow-up: "Why is the Zygote design important?" Without fork-from-Zygote, every app launch would have to re-load the entire Android framework (~200 MB worth of class metadata). Zygote sharing makes app launch sub-second.

Choreographer + VSYNC — how Android schedules frames

The display sends a VSYNC signal every ~16.67 ms (60 Hz) or 8.33 ms (120 Hz). The Choreographer sits on the main thread, listens for VSYNC, and dispatches three callbacks per frame: input → animation → traversal (measure + layout + draw). If a frame's work overruns one VSYNC interval, you drop a frame (jank).
// What Android does behind the scenes every frame:
Choreographer.getInstance().postFrameCallback { frameTimeNanos ->
    // 1. dispatch pending input events
    // 2. run animations (Animator updates, Compose recomposition)
    // 3. traverse views: measure → layout → draw
    // 4. swap buffers — display picks up the rendered frame at next VSYNC
}

  • RenderThread: GPU-side work (display lists) runs on a separate thread; main only builds the list, RenderThread dispatches to GPU.
  • Compose: hooks into Choreographer for recomposition timing; uses GraphicsLayer + RenderThread for transforms (no main-thread work).
  • JankStats library: detects dropped frames and reports them with a callback you can hook to analytics.

R8 deep dive: shrinking, optimization, obfuscation passes

R8 runs three passes. Shrinking removes unreferenced classes/methods/fields (tree-shaking from your -keep entry roots). Optimization inlines, propagates constants, devirtualizes, dead-code-eliminates. Obfuscation renames remaining classes/methods to short names. All three together typically cut APK size 30-60%.
  1. Tree shake: starting from -keep roots (your manifest entries, activities, R8 keep rules), walk the call graph; remove anything unreachable.
  2. Optimize: inline single-use methods, propagate constants, devirtualize calls (turn interface call into direct call when only one impl), eliminate unused branches, fold constant expressions.
  3. Obfuscate: rename com.app.repository.UserRepository.fetchUser → a.b.c.d. Stack traces become unreadable — hence the mapping.txt + Play Console upload.
  4. Outlining: identify code patterns repeated across many call sites and extract them into a shared method (saves bytes).

Modern Kotlin code benefits enormously: lambdas, suspend functions, data classes — R8 can inline a lot of this away.

Gotcha: Kotlin lambdas captured in lazy properties survive R8 inlining only if the property is also inlined. Sometimes you need to mark methods as @JvmStatic for R8 to optimize them away.

How does the Coroutine state machine compose? What happens on suspend?

The compiler converts each suspend function into a class implementing Continuation. Each suspension point is a label in a giant when. resumeWith jumps back to the next state. Composing two suspend functions = the inner state machine receives the outer's continuation as its callback.
suspend fun outer() {
    val a = inner1()        // suspend point #1
    val b = inner2(a)       // suspend point #2
    print(b)
}

// Compiles to roughly:
class outer$Continuation(val parent: Continuation<Unit>) : Continuation<Any?> {
    var label = 0
    var a: Any? = null
    override fun resumeWith(result: Result<Any?>) {
        when (label) {
            0 -> { label = 1; val r = inner1(this); if (r == COROUTINE_SUSPENDED) return; resumeWith(Result.success(r)) }
            1 -> { a = result.getOrThrow(); label = 2; val r = inner2(a, this); if (r == COROUTINE_SUSPENDED) return; resumeWith(Result.success(r)) }
            2 -> { print(result.getOrThrow()); parent.resumeWith(Result.success(Unit)) }
        }
    }
}

Suspending = the function returns COROUTINE_SUSPENDED immediately, having stored everything in the Continuation. When the awaited operation completes, IT calls resumeWith, which jumps to the next state. No threads blocked, no callbacks to write by hand.

Follow-up: "Why are coroutines almost free vs threads?" A continuation is ~hundreds of bytes; a thread stack is 1 MB. You can have millions of coroutines waiting for I/O on a handful of threads.

Binder IPC — how do two Android processes actually talk?

Binder is a kernel driver. Process A makes a call on a "proxy"; data is marshalled into a Parcel; the kernel switches to process B's "stub", which unmarshals and dispatches. Single-copy via shared memory (no intermediate kernel buffer for large data). Powers AIDL, ContentProvider cross-process queries, system services, and even ActivityManager (every startActivity is a Binder call to the system process).
  • IBinder: the interface. Every Service that returns a Binder from onBind exposes its API this way.
  • Parcel: a flat container of primitives + Parcelables for marshalling. Faster than Java serialization.
  • Shared memory: large payloads use ashmem or HardwareBuffer to avoid copy.
  • Death notifications: linkToDeath callbacks fire if the other process dies.
  • Limits: a Binder transaction Parcel can't exceed 1 MB system-wide — large data must be chunked or shared via ParcelFileDescriptor / file URI.

Most app code never writes a Binder directly — AIDL generates the proxy/stub. But every cross-process API in the OS (ActivityManagerService, WindowManagerService, ContentResolver) sits on Binder.

Follow-up: "Why is the 1 MB Binder limit so common in crash reports?" TransactionTooLargeException = Bundle/Parcel exceeded ~1 MB. Common when over-stuffing onSaveInstanceState. Fix: persist large state to disk (Room), keep Bundle small.

Bitmap memory: native heap, hardware bitmaps, ashmem

Pre-Android 8: bitmap pixel data lives on the Java heap (counts toward the per-app heap limit, ~256 MB). Android 8+: pixel data moved to native heap (ashmem / shared memory) — Java heap free of image bytes. Hardware bitmaps go even further: data lives in GPU memory, can't be read by CPU but draws faster.
  • ARGB_8888 bitmap: width × height × 4 bytes. A 4K (3840×2160) bitmap = 33 MB.
  • Android 8+: Bitmap.Config.ARGB_8888 stored on native heap. Java heap pressure relieved.
  • HARDWARE config: bitmap stored in GPU memory. No Java heap, no native heap. Display-only — pixel access throws.
  • BitmapPool: reuse Bitmap allocations across requests (Glide, Coil). Saves GC pressure.
  • inSampleSize: downsample at decode (BitmapFactory.Options.inSampleSize) — must be a power of 2.
  • Image library best practice: HARDWARE bitmap + memory cache for hot, disk LRU for warm, OkHttp + DiskCache for cold.

Modern image libraries (Glide, Coil) auto-pick HARDWARE on Android 8+ for display targets, falling back to ARGB_8888 if you do anything that requires pixel access (transformations, custom Painter).

Gotcha: A HARDWARE bitmap drawn into a software Canvas (e.g., to save a screenshot) silently re-decodes — slow path. Use copy(ARGB_8888, false) first.

How does ViewModel actually survive configuration changes?

Activity holds a NonConfigurationInstance field that the framework preserves across the destroy/create of a config change. Jetpack stuffs the ViewModelStore (a HashMap of ViewModels) in there. After rotation, the new Activity instance reads the same store — getting back the same ViewModel objects.
  1. Activity has onRetainNonConfigurationInstance() — a framework hook to keep arbitrary state across config-change destroy/create.
  2. Jetpack's ComponentActivity uses this to hold a NonConfigurationInstances object that contains the ViewModelStore.
  3. When the new Activity instance is constructed, Android passes the saved NonConfigurationInstance to it.
  4. ViewModelProvider(this).get(MyViewModel::class.java) looks up the store; finds the existing instance and returns it.
  5. If the Activity is finishing for real (not a config change), the store is cleared and onCleared() fires on each VM.
// What ComponentActivity does (simplified):
class ComponentActivity : Activity(), ViewModelStoreOwner {
    private var _store: ViewModelStore? = null
    override fun getViewModelStore() = _store ?: ViewModelStore().also { _store = it }

    override fun onRetainNonConfigurationInstance(): Any? {
        return NonConfigurationInstances(viewModelStore = _store)
    }

    override fun onCreate(b: Bundle?) {
        val nci = lastNonConfigurationInstance as? NonConfigurationInstances
        _store = nci?.viewModelStore        // reuse old store on rotation
    }
}

That's why onCleared() only fires on real finish, not rotation.

WorkManager internals — how it survives process death and schedules work

LiveData internals — how observers are notified lifecycle-aware

LiveData holds a list of ObserverWrappers, each pairing an observer with a LifecycleOwner. On setValue (main) or postValue (any thread → posts to main), it iterates wrappers, dispatches only to those whose lifecycle is STARTED or RESUMED. Wrapper auto-removes when lifecycle reaches DESTROYED.
// Roughly what LiveData does:
class LiveData<T> {
    private val observers = mutableListOf<ObserverWrapper<T>>()
    private var data: T? = null

    fun observe(owner: LifecycleOwner, obs: Observer<T>) {
        val w = LifecycleBoundObserverWrapper(owner, obs)
        observers += w
        owner.lifecycle.addObserver(w)         // auto-remove on DESTROYED
    }

    fun setValue(value: T) {
        assertMainThread()
        data = value
        observers.filter { it.shouldBeActive() }
            .forEach { it.observer.onChanged(value) }
    }

    fun postValue(value: T) {
        mainHandler.post { setValue(value) }
    }
}
  • Re-observing after a stop sends the latest value (sticky behavior).
  • postValue coalesces — if you call it twice rapidly, observers might only see the latest.
  • Lifecycle awareness is what makes LiveData safer than naive listeners — no dispatch to dead views.

SavedStateHandle — how it persists across process death

Each ViewModel can declare SavedStateHandle in its constructor. The library wires it to the Activity's SavedStateRegistry, which serializes all registered providers to a Bundle on onSaveInstanceState and restores them in onCreate(savedInstanceState). On process death, Bundle is what survives.
class EditViewModel(private val handle: SavedStateHandle) : ViewModel() {
    val draft: StateFlow<String> = handle.getStateFlow("draft", "")
    fun setDraft(t: String) { handle["draft"] = t }
}

Flow:

  1. SavedStateHandle is created when the ViewModel is constructed; it registers a SavedStateProvider on the Activity's SavedStateRegistry.
  2. On onSaveInstanceState, registry walks all providers, calls saveState() on each, packs into the outState Bundle.
  3. On process death, OS persists the outState Bundle.
  4. On relaunch, onCreate(savedInstanceState) hands the Bundle back; registry restores each provider's state.
  5. When the ViewModel is created (lazily), the SavedStateHandle reads its slice from the registry.

Bundle limit: ~1 MB system-wide. Don't put large data in SavedStateHandle — persist to Room and store just the ID in the handle.

Follow-up: "What types are supported?" Same as Bundle: primitives, Parcelable, Serializable, Bundle, ArrayList of primitives. Custom types must be Parcelable.

FCM internals — token persistence, message delivery, data vs notification payloads

Google Play Services holds a single persistent connection to FCM servers shared across all apps on the device. Your token is a routing identifier (per app+install). Server sends → FCM routes via the persistent socket → Play Services dispatches to your app's FirebaseMessagingService.
  1. Token issue: on first run, FirebaseInstallations registers; FCM issues a token.
  2. Token persistence: stored in Play Services data, not your app — survives app uninstalls? No, deletes with the install. Survives reboots, yes.
  3. Token rotation: onNewToken fires when token changes (clearing data, reinstall, server-side rotation). Always upload the new token.
  4. Single shared socket: Play Services maintains one connection to FCM (Google's infra). Your app doesn't open a socket.
  5. Server send: backend POSTs to fcm.googleapis.com/v1/projects/<id>/messages:send with target token + payload.
  6. Routing: FCM looks up which device owns that token and pushes via the persistent socket.
  7. Wakeup: Play Services invokes FirebaseMessagingService.onMessageReceived on your app (wakes process if needed).

Notification payload ("notification": {...}): Play Services renders the notification automatically when app is in background — your code doesn't run. Data payload ("data": {...}): your code always runs in onMessageReceived (foreground or background). Use this for full control + dedupe + custom render.

    override fun onNewToken(token: String) { uploadToBackend(token) }
    override fun onMessageReceived(msg: RemoteMessage) {
        val id = msg.data["id"] ?: return
        if (Dedupe.markSeen(id)) renderNotification(msg.data)
    }
}
Gotcha: Use data-only payloads in production — notification payloads silently bypass your code in background, breaking dedupe, deep linking, and custom rendering.

Compose custom Layout and SubcomposeLayout

Custom Layout = measure children once, then place. SubcomposeLayout = compose children based on the result of an earlier measurement (slower, but enables patterns like "slot the FAB above the snackbar"). Use SubcomposeLayout only when you can't avoid it.
// Custom Layout — vertical stack with custom spacing
@Composable fun SpacedColumn(spacing: Dp, content: @Composable () -> Unit) {
    Layout(content = content) { measurables, constraints ->
        val placeables = measurables.map { it.measure(constraints) }
        val spacingPx = spacing.roundToPx()
        val totalH = placeables.sumOf { it.height } + spacingPx * (placeables.size - 1)
        layout(constraints.maxWidth, totalH) {
            var y = 0
            placeables.forEach { it.placeRelative(0, y); y += it.height + spacingPx }
        }
    }
}

// SubcomposeLayout — used internally by Scaffold, BoxWithConstraints
SubcomposeLayout { constraints ->
    val headerPlaceable = subcompose("header") { Header() }
        .first().measure(constraints)
    val remaining = constraints.copy(maxHeight = constraints.maxHeight - headerPlaceable.height)
    val bodyPlaceable = subcompose("body") { Body() }
        .first().measure(remaining)
    layout(constraints.maxWidth, constraints.maxHeight) {
        headerPlaceable.placeRelative(0, 0)
        bodyPlaceable.placeRelative(0, headerPlaceable.height)
    }
}
Gotcha: SubcomposeLayout composes its children mid-measure — disables intrinsics and adds significant cost. Avoid if a regular Layout works.

Gradle dependency types: implementation vs api vs compileOnly vs runtimeOnly

    // Re-exports to consumers. Use ONLY for types in your public API.
    api("com.example:public-types:1.0")

    // Compile-time only (annotation processors at compile, missing at runtime)
    compileOnly("jakarta.annotation:jakarta.annotation-api:2.1")

    // Runtime-only (e.g., a logging implementation chosen at app level)
    runtimeOnly("ch.qos.logback:logback-classic:1.4")

    // KSP / kapt — annotation processors
    ksp("androidx.room:room-compiler:2.6.1")

    // Test scopes
    testImplementation("junit:junit:4.13.2")
    androidTestImplementation("androidx.test.ext:junit:1.1.5")
}
  • Prefer implementation by default — much faster incremental builds.
  • Switching from api to implementation on a heavily-used module can cut build time by 20-40%.
  • If a consumer module needs to use types from a transitive dep, that dep should be api in the providing module — or the consumer should declare it explicitly.

  • Compose has its own equivalent: painterResource reads VectorDrawable XML; rememberInfiniteTransition + Canvas for custom drawn shapes.

Q203Medium

Why Coroutines over RxJava — definitive answer

Coroutines: built into Kotlin, structured concurrency (no leaks), simpler mental model (write sync-style code), smaller binary, first-class Android lifecycle integration via Jetpack. RxJava: more operators, broader cross-platform, mature — but heavier and harder to teach.
  • Structured concurrency: parent scope cancels children automatically. Rx requires manual CompositeDisposable bookkeeping.
  • Compile-time safety: suspend functions can only be called from coroutines/suspend functions. Rx Subscriptions can be left dangling.
  • Code looks synchronous: easier debugging, simpler stack traces (with kotlinx-coroutines-debug).
  • Lifecycle integration: viewModelScope, lifecycleScope, repeatOnLifecycle are built-in. Rx needs AutoDispose / RxLifecycle.
  • APK size: Coroutines runtime ~20 KB; RxJava 3 ~1 MB.
  • RxJava advantages: more operators (1000+ vs Flow's ~80), schedulers more configurable, debounce/throttle/window operators very mature.

For new code: Coroutines + Flow. For legacy migration: incremental, use bridges (kotlinx-coroutines-rx3).

Q208Medium

Single-Activity vs Multi-Activity architecture

Single-Activity: one Activity hosting Compose Navigation. Cleaner deep links, shared element transitions, single source of truth for theme/insets/state. Multi-Activity: legacy default — more boilerplate, harder cross-screen state, but useful for genuinely separate experiences (auth flow, payment WebView, full-screen video).
  • Pros (Single-Activity): simpler Nav, shared transitions trivial, one ViewModelStore for app-wide state, unified theme/edge-to-edge.
  • Cons: navigation graph can become huge in big apps; entire app logic concentrated in one Activity.
  • Best of both: single-Activity in general, but launch a separate Activity for orthogonal experiences (3rd-party SDKs that demand it, immersive video player, OAuth WebView).

2026 default: single-Activity for new apps. Don't refactor a healthy multi-Activity app just for ideology.
