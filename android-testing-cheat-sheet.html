1. Why testing? — the problem and the cost of not testing

Tests aren't optional ceremony. They're the only way to keep a non-trivial codebase changeable without fear. Without tests:

  • Every refactor is a roll of the dice — you only find out you broke something when QA or users do
  • Onboarding new devs takes weeks because they're scared to touch anything
  • Bug fix → ship → new bug emerges in seemingly unrelated code (regression)
  • Performance work becomes guesswork instead of data-driven

The cost-of-bug curve

Where bug is caughtCost to fix
While writing the code (compiler / IDE)~1×
During unit tests on local machine~5×
During CI (PR review)~10×
During QA / staging~50×
In production (1-star reviews)~500× + reputation cost

Tests pull bug detection left on this curve. A good test suite turns a 500× incident into a 5× minor commit.

What good tests give you

  • Confidence to refactor — change implementation freely; tests verify behavior didn't break
  • Living documentation — tests show how a class is meant to be used (real usage, not stale comments)
  • Design feedback — hard-to-test code is usually badly designed code
  • Regression prevention — every fixed bug becomes a test that prevents it from coming back
  • Faster onboarding — new devs trust the safety net

3. Testing vocabulary — fakes, mocks, stubs, spies

TermWhat it meansWhen use
System under test (SUT)The class you're actually testingAlways the focus
FixtureThe setup data / state shared across testsJUnit's @BeforeEach
Test doubleUmbrella term for any stand-in (fake, mock, stub, spy, dummy)Whenever you replace a real dependency
DummyObject passed but never used (just to satisfy a parameter)Rare; mostly when you only care about other args
StubPre-programmed answers — "if asked, return X"When SUT only reads
MockStub + verifies interactions ("was method X called?")When SUT's correctness is "did it call this?"

FakeWorking implementation, simplified for tests (e.g., in-memory DB)Default choice for repositories and data sources

Fake vs Mock — the most important decision

Prefer FAKES for repository-like dependencies
  • Same interface as real, but uses an in-memory backing
  • Tests look like normal code — easy to read
  • Survives interface refactors (you only update one fake, not 50 mock setups)
  • Catches contract bugs (your fake exhibits same edge cases)
Use MOCKS sparingly — only for verification
  • Mocks tightly couple tests to implementation details
  • Refactor breaks 50 mock setups even if behavior is identical
  • Use only when "did the SUT call this method?" is the actual property under test (e.g., did Analytics.track fire?)
// FAKE — preferred for Repository
class FakeNotesRepository : NotesRepository {
    private val notes = mutableMapOf<String, Note>()
    private val flow = MutableStateFlow<List<Note>>(emptyList())
    override fun observe(): Flow<List<Note>> = flow
    override suspend fun add(note: Note) { notes[note.id] = note; flow.value = notes.values.toList() }
    override suspend fun get(id: String) = notes[id]
}

// MOCK — only when verifying interaction is the test's purpose
val analytics = mockk<Analytics>(relaxed = true)
vm.onClick()
verify { analytics.track("button_clicked") }      // the test IS about this call

5. Designing for testability — the 7 principles

  1. Depend on interfaces, not implementations. ViewModel takes UserRepository (interface), not UserRepositoryImpl. Lets you swap a fake.
  2. Inject dependencies through constructor. No val api = Retrofit.Builder()...build() inside classes — receive what you need.
  3. No hidden global state. Singletons that hold mutable state are test poison. Use Hilt scopes for true singletons; never object Manager { var x }.
  4. Inject the dispatcher. Don't hardcode Dispatchers.IO; receive it. Tests inject StandardTestDispatcher for virtual time.
  5. Inject the clock. Don't call System.currentTimeMillis() directly — wrap in a Clock interface. Tests inject a fixed clock for time-dependent logic.
  6. Pure functions wherever possible. Reducers, validators, formatters — all should be pure. Easiest tests in the world.
  7. Avoid static methods, top-level functions with side effects, and singletons that touch the OS. Wrap them in classes that can be replaced.
// Bad — untestable
class Repo {
    private val api = Retrofit.Builder()...build().create<Api>()    // hidden dep
    suspend fun get() = withContext(Dispatchers.IO) { api.fetch() }    // hidden dispatcher
    fun isExpired(token: Token) = token.expiresAt < System.currentTimeMillis()  // hidden clock
}

// Good — testable
class Repo @Inject constructor(
    private val api: Api,                       // inject interface
    private val ioDispatcher: CoroutineDispatcher, // inject dispatcher
    private val clock: Clock                    // inject clock
) {
    suspend fun get() = withContext(ioDispatcher) { api.fetch() }
    fun isExpired(token: Token) = token.expiresAt < clock.now()
}

6. The sample project — a Notes app we'll test layer by layer

For the rest of this guide we'll test a small Notes app. Here are the contracts (interfaces); implementations are obvious. Every later section tests one of these layers in isolation.

// Domain model
data class Note(val id: String, val title: String, val body: String, val updatedAt: Long)

// Repository interface
interface NotesRepository {
    fun observeAll(): Flow<List<Note>>
    suspend fun get(id: String): Note?
    suspend fun save(note: Note)
    suspend fun delete(id: String)
    suspend fun sync()
}

// UseCase
class SaveNoteUseCase @Inject constructor(
    private val repo: NotesRepository,
    private val clock: Clock
) {
    suspend operator fun invoke(title: String, body: String): Result<Note> {
        if (title.isBlank()) return Result.failure(IllegalArgumentException("empty title"))
        val note = Note(id = UUID.randomUUID().toString(), title = title, body = body, updatedAt = clock.now())
        repo.save(note)
        return Result.success(note)
    }
}

// ViewModel
@HiltViewModel
class NotesViewModel @Inject constructor(
    private val repo: NotesRepository,
    private val save: SaveNoteUseCase
) : ViewModel() {
    val notes: StateFlow<List<Note>> = repo.observeAll()
        .stateIn(viewModelScope, SharingStarted.WhileSubscribed(5_000), emptyList())

    private val _events = Channel<UiEvent>(Channel.BUFFERED)
    val events: Flow<UiEvent> = _events.receiveAsFlow()

    fun add(title: String, body: String) = viewModelScope.launch {
        save(title, body)
            .onSuccess { _events.send(UiEvent.Saved) }
            .onFailure { _events.send(UiEvent.Error(it.message ?: "unknown")) }
    }

    sealed interface UiEvent {
        data object Saved : UiEvent
        data class Error(val msg: String) : UiEvent
    }
}

// Composable
@Composable
fun NotesScreen(vm: NotesViewModel = hiltViewModel()) {
    val notes by vm.notes.collectAsStateWithLifecycle()
    Column {
        notes.forEach { Text(it.title, Modifier.testTag("note_${it.id}")) }
        Button(onClick = { vm.add("New", "Body") },
               modifier = Modifier.testTag("add_btn")) { Text("Add") }
    }
}

7. Test: plain Kotlin class

The simplest possible test. Pure logic, no dependencies, no Android. Use this every time you have a function that should be a pure function — extracted from somewhere bigger.

// Production class
class EmailValidator {
    fun isValid(email: String): Boolean =
        email.matches(Regex("^[A-Za-z0-9+_.-]+@[A-Za-z0-9.-]+\\.[A-Za-z]{2,}$"))
}

// JUnit 5 test
class EmailValidatorTest {
    private val validator = EmailValidator()

    @Test
    fun `accepts standard email`() {
        assertTrue(validator.isValid("sai@example.com"))
    }

    @Test
    fun `rejects email without @`() {
        assertFalse(validator.isValid("saiexample.com"))
    }

    @ParameterizedTest
    @ValueSource(strings = ["", "a@b", "@example.com", "a@.com"])
    fun `rejects invalid emails`(email: String) {
        assertFalse(validator.isValid(email))
    }
}

Best practices:

  • Use backticked function names — they read like sentences in test reports
  • One behavior per test (Arrange / Act / Assert pattern)
  • Use parameterized tests for "same logic, multiple inputs" — keeps test count up while code stays compact
  • Don't test private methods; test through public methods. If a private method is too complex to test that way, extract it into its own class with public methods

8. Test: UseCase (pure logic with injected deps)

UseCases are mostly orchestration: validate input, call the repo, format output. Inject everything; assert on Result.

class SaveNoteUseCaseTest {
    private val fakeRepo = FakeNotesRepository()
    private val fixedClock = object : Clock { override fun now() = 1_700_000_000_000L }
    private val useCase = SaveNoteUseCase(fakeRepo, fixedClock)

    @Test
    fun `saves note when title valid`() = runTest {
        val result = useCase("Title", "Body")
        assertTrue(result.isSuccess)
        val note = result.getOrThrow()
        assertEquals("Title", note.title)
        assertEquals(1_700_000_000_000L, note.updatedAt)
        assertEquals(1, fakeRepo.allSnapshot().size)
    }

    @Test
    fun `fails when title blank`() = runTest {
        val result = useCase("", "body")
        assertTrue(result.isFailure)
        assertTrue(result.exceptionOrNull() is IllegalArgumentException)
        assertEquals(0, fakeRepo.allSnapshot().size)
    }
}

// FakeNotesRepository
class FakeNotesRepository : NotesRepository {
    private val store = mutableMapOf<String, Note>()
    private val flow = MutableStateFlow<List<Note>>(emptyList())

    override fun observeAll(): Flow<List<Note>> = flow
    override suspend fun get(id: String) = store[id]
    override suspend fun save(note: Note) { store[note.id] = note; flow.value = store.values.toList() }
    override suspend fun delete(id: String) { store.remove(id); flow.value = store.values.toList() }
    override suspend fun sync() { /* no-op for tests */ }

    // Test-only helpers
    fun allSnapshot(): List<Note> = store.values.toList()
    fun seed(notes: List<Note>) { notes.forEach { store[it.id] = it }; flow.value = store.values.toList() }
}

9. Test: Repository (with fake DAO + fake Api)

Repositories orchestrate data sources. Test "given network returns X, repo emits Y" — without spinning up Room or OkHttp.

class NotesRepositoryImplTest {
    private val fakeDao = FakeNotesDao()
    private val fakeApi = FakeNotesApi()
    private val repo = NotesRepositoryImpl(fakeDao, fakeApi)

    @Test
    fun `sync writes remote results to DAO`() = runTest {
        fakeApi.notesToReturn = listOf(Note("1", "a", "", 0), Note("2", "b", "", 0))
        repo.sync()
        assertEquals(2, fakeDao.allSnapshot().size)
    }

    @Test
    fun `observeAll emits from DAO`() = runTest {
        fakeDao.seed(listOf(Note("x", "x", "", 0)))
        repo.observeAll().test {
            assertEquals(1, awaitItem().size)
            cancelAndIgnoreRemainingEvents()
        }
    }

    @Test
    fun `save persists locally and queues remote sync`() = runTest {
        val note = Note("1", "t", "b", 0)
        repo.save(note)
        assertTrue(fakeDao.allSnapshot().contains(note))
        assertTrue(fakeApi.savedNotes.contains(note))
    }
}

class FakeNotesDao : NotesDao {
    private val store = mutableMapOf<String, Note>()
    private val flow = MutableStateFlow<List<Note>>(emptyList())
    override fun observeAll() = flow
    override suspend fun upsert(note: Note) { store[note.id] = note; flow.value = store.values.toList() }
    override suspend fun get(id: String) = store[id]
    override suspend fun delete(id: String) { store.remove(id); flow.value = store.values.toList() }
    fun allSnapshot() = store.values.toList()
    fun seed(notes: List<Note>) { notes.forEach { store[it.id] = it }; flow.value = store.values.toList() }
}

class FakeNotesApi : NotesApi {
    var notesToReturn: List<Note> = emptyList()
    val savedNotes = mutableListOf<Note>()
    override suspend fun fetchAll() = notesToReturn
    override suspend fun push(note: Note) { savedNotes += note }
}

10. Test: ViewModel (Coroutines + StateFlow + SharedFlow events)

The most-tested layer in a typical project. Three things to verify: state evolves correctly, events fire correctly, no leaks.

@OptIn(ExperimentalCoroutinesApi::class)
class NotesViewModelTest {

    private val dispatcher = StandardTestDispatcher()
    private val fakeRepo = FakeNotesRepository()
    private val useCase = SaveNoteUseCase(fakeRepo, FixedClock(100L))
    private lateinit var vm: NotesViewModel

    @BeforeEach
    fun setUp() {
        Dispatchers.setMain(dispatcher)        // viewModelScope uses Main
        vm = NotesViewModel(fakeRepo, useCase)
    }

    @AfterEach
    fun tearDown() {
        Dispatchers.resetMain()
    }

    @Test
    fun `notes state mirrors repository`() = runTest(dispatcher) {
        vm.notes.test {
            assertEquals(emptyList<Note>(), awaitItem())               // initial
            fakeRepo.seed(listOf(Note("1", "a", "", 0)))
            assertEquals(1, awaitItem().size)
            cancelAndIgnoreRemainingEvents()
        }
    }

    @Test
    fun `add success emits Saved event and persists`() = runTest(dispatcher) {
        vm.events.test {
            vm.add("Title", "Body")
            advanceUntilIdle()                                          // drain pending coroutines
            assertEquals(NotesViewModel.UiEvent.Saved, awaitItem())
        }
        assertEquals(1, fakeRepo.allSnapshot().size)
    }

    @Test
    fun `add empty title emits Error event`() = runTest(dispatcher) {
        vm.events.test {
            vm.add("", "body")
            advanceUntilIdle()
            val evt = awaitItem()
            assertTrue(evt is NotesViewModel.UiEvent.Error)
        }
        assertEquals(0, fakeRepo.allSnapshot().size)
    }
}
Three patterns to memorize:
  • Dispatchers.setMain(dispatcher) in setUp + resetMain() in tearDown — required because viewModelScope uses Dispatchers.Main which is unavailable in JVM tests by default.
  • runTest(dispatcher) + advanceUntilIdle() — drives coroutines deterministically.
  • flow.test { awaitItem() } — Turbine; the cleanest way to assert Flow emissions.

11. Test: Room DAO (in-memory database, fast)

Room DAO tests need a real Room — the generated implementations are non-trivial. Use an in-memory database in androidTest/. With Robolectric you can also run Room on the JVM for some scenarios, but instrumented in-memory is the canonical approach.

@RunWith(AndroidJUnit4::class)
class NotesDaoTest {
    private lateinit var db: AppDatabase
    private lateinit var dao: NotesDao

    @Before
    fun setUp() {
        val ctx = ApplicationProvider.getApplicationContext<Context>()
        db = Room.inMemoryDatabaseBuilder(ctx, AppDatabase::class.java)
            .allowMainThreadQueries()
            .build()
        dao = db.notesDao()
    }

    @After fun tearDown() = db.close()

    @Test fun insertAndGet() = runTest {
        val note = Note("1", "t", "b", 100)
        dao.upsert(note)
        assertEquals(note, dao.get("1"))
    }

    @Test fun observeAllEmitsOnInsert() = runTest {
        dao.observeAll().test {
            assertEquals(emptyList<Note>(), awaitItem())
            dao.upsert(Note("1", "t", "", 0))
            assertEquals(1, awaitItem().size)
            cancelAndIgnoreRemainingEvents()
        }
    }
}

12. Test: Retrofit Api (MockWebServer)

Don't mock Retrofit. Use a real OkHttp client pointed at a local MockWebServer — same code path as production.

class NotesApiTest {
    private lateinit var server: MockWebServer
    private lateinit var api: NotesApi

    @BeforeEach fun setUp() {
        server = MockWebServer().apply { start() }
        api = Retrofit.Builder()
            .baseUrl(server.url("/"))
            .addConverterFactory(MoshiConverterFactory.create())
            .build()
            .create(NotesApi::class.java)
    }

    @AfterEach fun tearDown() = server.shutdown()

    @Test fun `fetchAll parses successful response`() = runTest {
        server.enqueue(MockResponse().setBody("""
            [{"id":"1","title":"Hello","body":"World","updatedAt":100}]
        """.trimIndent()))

        val notes = api.fetchAll()
        assertEquals(1, notes.size)
        assertEquals("Hello", notes.first().title)

        val recorded = server.takeRequest()
        assertEquals("GET", recorded.method)
        assertEquals("/notes", recorded.path)
    }

    @Test fun `fetchAll throws HttpException on 500`() = runTest {
        server.enqueue(MockResponse().setResponseCode(500))
        assertThrows<HttpException> { api.fetchAll() }
    }
}

13. Test: Composable (createComposeRule)

Compose UI tests are fast (no view inflation), instrumented, and very expressive. Use testTag for stable selectors.

@RunWith(AndroidJUnit4::class)
class NotesScreenTest {

    @get:Rule val compose = createComposeRule()

    @Test fun `clicking add invokes callback`() {
        var addCalled = false
        compose.setContent {
            NotesScreen(
                state = NotesScreenState(notes = emptyList()),
                onAdd = { addCalled = true }
            )
        }
        compose.onNodeWithTag("add_btn").performClick()
        assertTrue(addCalled)
    }

    @Test fun `renders all notes`() {
        compose.setContent {
            NotesScreen(
                state = NotesScreenState(notes = listOf(
                    Note("1", "First", "", 0),
                    Note("2", "Second", "", 0)
                )),
                onAdd = {}
            )
        }
        compose.onNodeWithText("First").assertIsDisplayed()
        compose.onNodeWithText("Second").assertIsDisplayed()
        compose.onAllNodesWithTag("note_", useUnmergedTree = true).assertCountEquals(2)
    }
}

Pattern: hoist state out of the Composable so tests pass static state directly. The above NotesScreen(state, onAdd) is a stateless variant; the full NotesScreen() with ViewModel calls into this stateless one. This separation makes UI tests trivial.

Key APIs:

  • onNodeWithText("X") / onNodeWithTag("tag") / onNodeWithContentDescription("desc")
  • performClick() / performTextInput("...") / performScrollTo()
  • assertIsDisplayed() / assertIsEnabled() / assertTextEquals("...")
  • compose.waitForIdle() — wait for animations / async work
  • compose.mainClock.autoAdvance = false; advanceTimeBy(ms) — control time for animation tests

15. Test: ContentProvider

Use the ProviderTestRule in instrumented tests. It sets up an isolated MockContentResolver pointing at your provider — no real provider registration needed.

@RunWith(AndroidJUnit4::class)
class NotesProviderTest {
    @get:Rule val providerRule = ProviderTestRule.Builder(
        NotesProvider::class.java,
        "com.example.notes.provider"
    ).build()

    @Test fun `insert and query round-trip`() {
        val resolver = providerRule.resolver
        val uri = Uri.parse("content://com.example.notes.provider/notes")
        val values = ContentValues().apply {
            put("id", "1"); put("title", "t"); put("body", "b")
        }
        val inserted = resolver.insert(uri, values)
        assertNotNull(inserted)

        resolver.query(uri, null, null, null, null)?.use { c ->
            assertTrue(c.moveToFirst())
            assertEquals("t", c.getString(c.getColumnIndexOrThrow("title")))
        }
    }
}

Better pattern: wrap your ContentProvider behind a Repository / DataSource interface. Then your application code talks to the interface, and you can test with a fake — same pattern as Retrofit / Room.

16. Test: SharedPreferences / DataStore

For new code, use DataStore (preferred). Both can be tested cleanly.

SharedPreferences (legacy)

@RunWith(RobolectricTestRunner::class)
class SettingsRepositorySharedPrefsTest {
    private lateinit var ctx: Context
    private lateinit var repo: SettingsRepository

    @Before fun setUp() {
        ctx = ApplicationProvider.getApplicationContext()
        repo = SettingsRepository(ctx.getSharedPreferences("test", Context.MODE_PRIVATE))
    }

    @Test fun `theme persists across reads`() {
        repo.setTheme("dark")
        assertEquals("dark", repo.theme())
    }
}

DataStore (preferred)

class SettingsDataStoreTest {
    @TempDir lateinit var tempDir: File
    private lateinit var dataStore: DataStore<Preferences>
    private lateinit var repo: SettingsRepository

    @BeforeEach fun setUp() {
        dataStore = PreferenceDataStoreFactory.create(
            scope = TestScope(),
            produceFile = { File(tempDir, "settings.preferences_pb") }
        )
        repo = SettingsRepository(dataStore)
    }

    @Test fun `theme write then read`() = runTest {
        repo.setTheme("dark")
        assertEquals("dark", repo.themeFlow().first())
    }
}

Key trick: create the DataStore with a temporary directory so each test has fresh state and there's no file conflict between tests running in parallel.

19. Coroutines & Flow testing — deep dive

The 3 helpers you'll use everywhere

HelperWhat it does
runTest { }Run suspending test code with virtual time. Replaces runBlocking in tests.
StandardTestDispatcherDoesn't auto-run coroutines; you call runCurrent() / advanceTimeBy(ms) / advanceUntilIdle(). Use when timing matters.
UnconfinedTestDispatcherAuto-runs each coroutine eagerly until first suspension. Easier but less precise.
 
@Test fun `debounce + flatMapLatest cancels prior request`() = runTest {
    val queries = MutableSharedFlow<String>(replay = 0)
    val calls = mutableListOf<String>()

    val job = launch {
        queries
            .debounce(300)
            .flatMapLatest { q -> flow { calls += q; emit("result $q") } }
            .collect()
    }

    queries.emit("a")
    advanceTimeBy(100)                    // not yet past debounce
    queries.emit("ab")                    // resets debounce timer
    advanceTimeBy(300)                    // past debounce since last emit
    runCurrent()

    assertEquals(listOf("ab"), calls)     // only the latest reached the producer
    job.cancel()
}

Turbine for Flow assertions

flow.test {
    assertEquals(1, awaitItem())
    assertEquals(2, awaitItem())
    awaitComplete()                              // or cancelAndIgnoreRemainingEvents()
}

// Wait with timeout
flow.test(timeout = 2.seconds) {
    expectMostRecentItem()                       // drains buffered items, returns latest
}

20. Screenshot testing — Paparazzi (View) and Roborazzi (Compose)

Render UI to PNG without an emulator and diff against a "golden" image. Catches design regressions instantly. Cheap to run on CI.

class NotesScreenScreenshotTest {

22. Macrobenchmark — measuring performance

For perf regressions (cold start, scroll jank, key-flow timing), use Macrobenchmark — runs against a release build on a real device.

@RunWith(AndroidJUnit4::class)
class StartupBenchmark {

    @Test fun coldStartup() = rule.measureRepeated(
        packageName = "com.example.notes",

        packageName = "com.example.notes",

23. Code coverage — Kover (KMP) & JaCoCo (JVM)

// build.gradle.kts

    id("org.jetbrains.kotlinx.kover") version "0.7.6"

Coverage trap: 100% coverage doesn't mean 100% tested. A test that calls a function and asserts nothing scores full coverage and tests nothing. Use coverage as a floor, not a goal.

24. Common mistakes & anti-patterns

Don't
  • Test private methods directly (test through public)
  • Mock everything (over-coupled to implementation)
  • Use Thread.sleep in coroutine tests (use virtual time)
  • Share mutable fixtures across tests (flaky test cascade)
  • Test framework code (Room generated class is Google's responsibility)
  • Write tests that depend on test order (each test must be independent)
  • Use runBlocking in tests (use runTest)
  • Hit real network in unit tests (use MockWebServer)
  • Skip writing tests because "I'll add them later" (you won't)
  • Pursue 100% coverage (diminishing returns; flaky-test risk)
  • Mock data classes (just construct them with values)
  • Mock final classes when a fake works (Kotlin classes are final by default — fakes are usually cleaner)
Do
  • Hoist state out of Composables — makes UI tests trivial
  • Inject dispatchers and clock
  • Use fakes for repositories, mocks only for verification
  • Run tests on every PR via CI
  • Treat flaky tests as bugs — fix or delete, never ignore
  • Write the test before the bug fix (proves the fix works)
  • Use @TempDir for any file-based test fixtures
  • Name tests as full sentences — they're documentation
  • Group related tests in nested classes (JUnit 5 @Nested)
  • Use parameterized tests for "same logic, many inputs"
  • Profile slow tests; aim for unit tests under 100 ms each

25. Interview Q&A — drill answers

Q: "Fakes vs mocks — which do you prefer and why?"

A: Fakes for almost everything that has substantial behavior (Repository, DataStore, Api). Mocks only when verifying interaction is the actual property under test (analytics, logger). Reasons: fakes survive refactors; tests look like normal code; hard-to-test contracts get caught (your fake exhibits the same edge cases). Mocks tightly couple tests to implementation; refactor breaks 50 mock setups.

Q: "How do you test a ViewModel that uses Coroutines and Flow?"

A: Three pieces. (1) Dispatchers.setMain(StandardTestDispatcher()) in setUp + resetMain() in tearDown — viewModelScope uses Dispatchers.Main which is unavailable in JVM tests by default. (2) Wrap the test body in runTest(dispatcher) for virtual time + advanceUntilIdle() to drain. (3) Use Turbine (flow.test { awaitItem() }) for asserting Flow emissions sequentially.

Q: "How do you test a Repository?"

A: Inject fake DAO and fake Api into the real Repository implementation. Verify the data orchestration logic (e.g., "sync writes API results to DAO", "save persists locally and queues remote"). For DAO/Api themselves, separate tests — DAO with in-memory Room (instrumented), Api with MockWebServer.

Q: "How do you test a Composable?"

A: Hoist state out — make it a stateless Composable taking (state, callbacks). Then createComposeRule(), setContent { MyScreen(state, callback) }, onNodeWithText/Tag for selectors, performClick/performTextInput for interactions, assertIsDisplayed/assertTextEquals for assertions. The full screen with ViewModel just calls into the stateless one — separate Hilt integration test for the wiring.

Q: "When would you use Robolectric vs instrumented tests?"

A: Robolectric runs Android API on JVM (fast, no emulator). Use for: SharedPreferences/Service/legacy Android API tests where you want speed. Instrumented tests run on a real device/emulator (slow but accurate). Use for: Room DAO (in-memory database), Compose UI tests, anything that needs real GL / hardware acceleration / system services beyond Robolectric's shadows.

Q: "What's the testing pyramid and why does shape matter?"

A: Many fast cheap unit tests at the base, fewer integration tests in the middle, very few slow expensive UI tests at the top. Shape matters because UI tests are 10× slower and 10× flakier than unit tests for the same coverage. Inverted pyramids (many UI tests, few unit tests) make CI slow, flaky, and expensive — bug detection happens late on the cost curve.

Q: "How do you write testable code?"

A: Five things: (1) constructor injection, never val x = SomeStatic.create(); (2) depend on interfaces, not implementations; (3) inject dispatcher and clock; (4) keep functions pure where possible; (5) hoist state in Compose. The pattern: every external dependency is a parameter; every parameter can be a fake.

Q: "Coroutine test — why runTest and not runBlocking?"

A: runTest uses virtual time — delay(60_000) resolves immediately. Tests stay fast. runBlocking blocks real threads, makes tests slow and timing-sensitive. Use runTest for any test that touches coroutines, runBlocking only for bridging to legacy non-suspend code in test setup.

Q: "Why prefer MockK over Mockito for Kotlin?"

A: MockK supports Kotlin features Mockito struggles with: final classes by default (Kotlin classes are final unless marked open), suspend functions (coEvery/coVerify), top-level functions, object/companion mocking, lambda captures. With Mockito you need mockito-inline for finals + manual setup for coroutines.

Q: "How do you test Hilt?"

A: Annotate test class @HiltAndroidTest, add HiltAndroidRule, call hilt.inject() in @Before. Replace production modules in tests via @TestInstallIn(replaces = [DataModule::class]) on a fake module. Use a custom Application for instrumented tests — generate via @CustomTestApplication.

Q: "How do you test code that depends on time?"

A: Inject a Clock interface (fun now(): Long). Production: real Clock returning System.currentTimeMillis(). Tests: FixedClock(100L) returning a known value. Same trick for any system call — wrap behind an interface, inject, override in tests.

Q: "Should we test private methods?"

A: No. Test through public methods; if a private method is too complex to be exercised that way, extract it to its own class with public methods. Testing privates couples tests to implementation; you'll be unable to refactor without rewriting tests.

Q: "What's a flaky test? How do you handle it?"

A: A test that sometimes passes, sometimes fails with no code change. Causes: timing dependencies, shared state between tests, race conditions, network. Treat flaky tests as bugs — fix or delete, never ignore. Common fixes: use virtual time (runTest), isolate fixtures (@TempDir), use stable selectors (testTag), eliminate Thread.sleep.

26. One-page recall card

The mental model

UI = f(state). Test logic in isolation (unit), test wiring across layers (integration), test critical user journeys (E2E). Push every test as low on the pyramid as possible. Design for testability: inject everything, depend on interfaces.

The latest stack (2026)

  • JUnit 5 (JVM) / Kotest (KMP)
  • MockK (mocks) + hand-written fakes (preferred for repos)
  • kotlinx-coroutines-test + Turbine
  • MockWebServer for HTTP
  • Room in-memory for DAO
  • Robolectric for Android API on JVM
  • Compose Test rule for UI
  • Hilt testing for DI integration
  • Paparazzi / Roborazzi for screenshots
