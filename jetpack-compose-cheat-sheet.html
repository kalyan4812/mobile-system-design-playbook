1. The mental model — Compose in 60 seconds

Jetpack Compose is a declarative Kotlin-DSL UI toolkit. You describe what the UI should look like for a given state; Compose figures out how to update the screen when state changes.

Old View system (XML + Kotlin/Java)Jetpack Compose
StyleImperative: mutate views directlyDeclarative: UI = f(state)
Source of UI definitionXML + code merging via findViewById/bindingOne Kotlin file with @Composable functions
State syncManual (you push state into views, easy to drift)Automatic (state change → recomposition)
Custom UICustom View / ViewGroup (heavy)Custom Composable + Layout (light)
AnimationAnimator + ObjectAnimatoranimate*AsState, Transition, integrated
Live previewLayout Inspector after launch@Preview in Android Studio, instant

Three things to internalize:

  1. UI is a function of state. Same state in → same UI out. Never mutate widgets directly.
  2. Recomposition is cheap, not free. Compose reruns Composable functions when state they read changes. The runtime tries to skip subtrees that haven't changed (smart recomposition), but writing skippable code is on you.
  3. Composables are short-lived. The function body re-runs many times. Side effects must go in side-effect APIs, not the function body itself.
@Composable
fun Greeting(name: String) {
    Text("Hello, $name!")
}
// Compose runtime tracks the read of `name`. Any change → just this Composable re-runs.

2. Composable functions — what @Composable actually does

@Composable is not a marker. It changes the function's type and contract. After the Compose Compiler plugin runs, your function silently gets a hidden Composer parameter, plus extra bookkeeping parameters (more in §6 internals).

// You write:
@Composable
fun NamePlate(name: String) { Text(name) }

// Compiler emits roughly:
fun NamePlate(name: String, $composer: Composer, $changed: Int) { ... }

That hidden Composer parameter is what makes the magic work. Composables can only be called from other Composables (or special entry points like setContent) — because only then is a Composer in scope to hand off.

The five rules of well-behaved Composable functions

RuleWhat it meansWhy it matters
Calling contextCan only be invoked where a Composer is in scopeThe runtime needs the Composer to track this call's position in the tree
IdempotentSame inputs produce same effect on the tree, every timeSo Compose can skip / replay safely
Free of side effectsThe body itself doesn't observably change app state outside ComposeSide effects belong in LaunchedEffect / DisposableEffect / SideEffect
RestartableCompose may re-run the function many times when its inputs changeYour code must tolerate repeated execution without breaking
FastBody should be cheap (microseconds), not do real workRecomposition happens on the UI thread; slow Composables = jank

Bonus rule: Composables are also positionally memoized — Compose tracks each call's position in the tree, so two calls to the same Composable at different positions are independent.

Composable functions are "colored" — like suspend

Just like suspend, @Composable partitions the function space. A regular function can't call a Composable. A Composable can only be called from another Composable (or a builder like setContent { } or composable { }). This is enforced at compile time. Same idea as Kotlin's "colored" suspend functions.

3. State — mutableStateOf, remember, rememberSaveable

State is what makes recomposition meaningful. Compose tracks every State read inside Composables; when that State's value changes, the readers are scheduled for recomposition.

@Composable
fun Counter() {
    var count by remember { mutableStateOf(0) }   // State of Int
    Button(onClick = { count++ }) {                  // write — State updates
        Text("Clicks: $count")                       // read — this Text recomposes when count changes
    }
}
HelperSurvives recomposition?Survives config change?Survives process death?
remember { ... }YesNoNo
rememberSaveable { ... }YesYes (via Bundle)Yes (via Bundle)
State in ViewModelYesYesNo (use SavedStateHandle for that)

The four State factories you'll use

  • mutableStateOf(initial) — basic MutableState<T>. Use by delegate for property-style access.
  • mutableIntStateOf(0) / mutableLongStateOf / etc. — primitive variants, avoid boxing in hot paths (lists, animations).
  • mutableStateListOf() / mutableStateMapOf() — observable lists/maps; mutating them triggers recomposition of readers.
  • derivedStateOf { computed } — wraps a derived value so recomposition fires only when the result actually changes (see §10).
Don't: use plain var x = 0 inside a Composable for mutable UI state. Updating it does not trigger recomposition. Always wrap mutable UI state in State.

What by actually does

var count by remember { mutableStateOf(0) } uses Kotlin property delegation. The delegate's getValue reads state.value, the setValue writes it. So count++ compiles to state.value = state.value + 1. Pure syntactic sugar — the State object is still what Compose tracks.

4. Recomposition — what it is, when it happens

Recomposition = re-running a Composable function with new inputs. It happens because something the Composable read (a State value, a CompositionLocal, or one of its parameters) changed.

What gets recomposed?

  • Compose tracks which Composable read which State.
  • When State changes, Compose schedules the smallest enclosing RecomposeScope (typically the nearest restartable Composable function) for re-execution.
  • Other parts of the tree are skipped if their inputs haven't changed and their parameters are stable (see §9 for stability).
@Composable
fun Counter() {
    var a by remember { mutableStateOf(0) }
    var b by remember { mutableStateOf(0) }
    Column {
        Text("A: $a")            // reads a → only this Text re-runs when a changes
        Text("B: $b")            // reads b → only this Text re-runs when b changes
        Button(onClick = { a++ }) { Text("+A") }
        Button(onClick = { b++ }) { Text("+B") }
    }
}

Tapping +A only re-runs the first Text. The Column, Buttons, and the second Text are skipped. This is the heart of "smart recomposition."

Tip — Layout Inspector: turn on "Show recomposition counts" in Android Studio's Layout Inspector to see exactly which Composables recompose, how often, and how many were skipped. Best perf-debugging tool you have.

5. State hoisting — the most important pattern

"Hoist state" = move state out of a Composable to its caller. The Composable becomes stateless and accepts (value, onValueChange) as parameters. Now it's reusable, testable, previewable, and the caller controls behavior.

// Bad: state hidden inside, no way for caller to read or react
@Composable
fun NameInput() {
    var name by remember { mutableStateOf("") }
    TextField(name, onValueChange = { name = it })
}

// Good: hoisted
@Composable
fun NameInput(name: String, onNameChange: (String) -> Unit) {
    TextField(name, onValueChange = onNameChange)
}

// Caller owns state — can validate, persist, share with other Composables
var name by rememberSaveable { mutableStateOf("") }
NameInput(name, onNameChange = { name = it.trim() })

Where should state live?

  • UI-only ephemeral (focus state, expanded/collapsed, scroll position) → in the Composable via remember
  • Survives rotation but not process death → ViewModel (StateFlow + collectAsStateWithLifecycle)
  • Survives process death → rememberSaveable for small primitives, SavedStateHandle in ViewModel for VM-scoped, Room/DataStore for durable
  • Shared across Composables → hoist to lowest common ancestor; for app-wide, ViewModel

Rule of thumb: hoist state to the lowest common ancestor of all Composables that need it. Don't hoist higher than necessary (creates unnecessary recompositions).

★ Internals — what's NOT magic (the engine demystified)

People treat Compose as magic. It isn't. Every "magic" behavior is a small piece of plain Kotlin code. Once you understand the engine, the framework's quirks make sense and you stop fighting it.

The three big building blocks

  1. Compose Compiler — a Kotlin compiler plugin that rewrites every @Composable function to inject the Composer and add tracking metadata.
  2. Compose Runtime — the in-memory data structure (the slot table) that records what was composed, plus the Composer that manages reading/writing it.
  3. Snapshot System — the read-tracking mechanism that knows which Composable read which State, so changes can invalidate the right RecomposeScopes.

1. What the Compose Compiler does to your @Composable function

Looks like magic: annotating a function with @Composable somehow adds Compose superpowers.
Actually: the Compose Compiler plugin transforms each Composable in three ways:
  1. Injects hidden parameters: a Composer instance + a $changed: Int bitmask + a $default: Int bitmask (for default arguments).
  2. Wraps the body in composer.startRestartGroup(key) and composer.endRestartGroup()?.updateScope { ... }. This creates a RecomposeScope — the unit Compose can re-execute on its own.
  3. Adds skip logic: at the top of the function, check if all parameters are unchanged AND stable. If yes, call composer.skipToGroupEnd() and return without running the body — that's recomposition skipping.
// You write:
@Composable fun Greeting(name: String) { Text("Hello $name") }

// Compiler emits roughly (simplified):
fun Greeting(name: String, $composer: Composer, $changed: Int) {
    $composer.startRestartGroup(0xABCD)
    var dirty = $changed
    if ($changed and 0b1 == 0)
        dirty = dirty or if ($composer.changed(name)) 0b100 else 0b10

    if (dirty and 0b11 != 0b10 || !$composer.skipping) {
        Text("Hello $name", $composer, 0)
    } else {
        $composer.skipToGroupEnd()                // ← skip!
    }

    $composer.endRestartGroup()?.updateScope { c, _ ->
        Greeting(name, c, $changed or 1)        // how to re-run if invalidated
    }
}

2. The slot table — Compose's in-memory tree

Looks like magic: Compose somehow remembers state across recompositions even though Composables are just function calls.
Actually: the Composer maintains a slot table, which is a flat array (gap buffer) representing the tree of all Composable calls and their stored values. Every remember, every State, every parameter, every group boundary — they all become slots in this array.

How a Composable reads/writes the slot table:

  • On first composition, Compose writes a new entry: a group token, the parameters, the remembered values.
  • On recomposition, Compose reads the entry at the same position, comparing parameters. If unchanged, skip. If changed, update.
  • The flat-array layout means traversal is cache-friendly. The "gap buffer" lets Compose insert/remove items in the middle efficiently.

3. Positional memoization — why remember works

Looks like magic: calling remember { generateExpensiveValue() } twice doesn't run the lambda twice — even though it's a normal Kotlin function call.
Actually: remember stores its result at the current position in the slot table. On re-execution, it asks the Composer "is there already a value at this position with these inputs?" If yes, returns it without invoking the lambda. The "position" is determined by the call site in the source code (Composer's source-location tracking), not by argument identity. That's why two remember calls in the same function produce two independent memoized values. 
@Composable fun Demo() {
    val a = remember { compute(1) }    // stored at position P1
    val b = remember { compute(2) }    // stored at position P2
    // Even if compute(1) == compute(2), these are DIFFERENT memoized slots.
}

4. The snapshot system — how reads of State get tracked

Looks like magic: reading state.value inside a Composable somehow registers that "this Composable depends on this State."
Actually: when Compose enters a RecomposeScope, it sets a thread-local SnapshotObserver. Every state.value read goes through the State's readable method, which calls back to the observer: "scope X read state Y." The observer records this in a (state → scopes) map. When state.value is later set to a new value, the observer fires the recomposition for every scope that read it.

This is why Composable functions must read State during their execution — not later, not in a callback. If the read happens outside the observer's context, it's not tracked, and recomposition won't trigger.

5. The Snapshot itself — MVCC for State

Snapshots aren't just a tracking mechanism. Each Snapshot is a versioned view of all State at a point in time. This enables:

  • Concurrent reads: composition can read State while another thread is mutating, without locks — each side sees a consistent snapshot.
  • Atomic mutation: Snapshot.takeMutableSnapshot() lets you mutate State in isolation, then atomically apply() all changes together (or abandon them).
  • Conflict detection: when applying, Compose checks for write conflicts and can merge/reject.

It's basically Multi-Version Concurrency Control (MVCC), the same idea databases use.

6. Smart Recomposition — why most of the tree is skipped

Looks like magic: only the affected Composable re-runs; thousands of others are skipped.
Actually: three things combine.
  1. Per-State subscriber tracking (above) → Compose knows exactly which RecomposeScopes depend on the changed State.
  2. Per-parameter change check via $changed bitmask → on entry to a Composable, Compose compares old vs new parameters bit-by-bit; unchanged + stable means "skip the body."
  3. Slot-table replay → for skipped groups, Compose just advances the slot table pointer without executing — the values are already there.

7. What is a RecomposeScope?

A RecomposeScope is the smallest unit of Composition that Compose can re-run independently. By default, every restartable Composable function is its own scope. Inline functions and lambdas with the @Composable annotation may or may not get their own scope (the compiler decides based on whether they have parameters and whether they're stable).

Practical impact: if you read State inside a Composable, the smallest enclosing scope is invalidated. So defer reads as far down the tree as possible — read inside a leaf, not the root.

8. Lambdas inside Composables get memoized

Looks like magic: passing onClick = { vm.click() } doesn't cause infinite recompositions even though a new lambda is created on every call.
Actually: the Compose Compiler does lambda memoization. If a lambda has no captured variables, or all its captures are stable and unchanged, the compiler wraps it in a remember-equivalent slot and reuses the same instance. So your child Composable receives the same lambda reference and stays skippable.

9. The Recomposer — the orchestrator

The Recomposer is a singleton coroutine running on the AndroidUiDispatcher (Compose's UI dispatcher). It receives invalidation signals from State changes, batches them per frame, and triggers recomposition on the affected scopes when the next frame is requested. This is why state changes are coalesced — multiple mutations in the same frame produce a single recomposition.

10. setContent { } — what actually happens at the entry point

  • An AbstractComposeView is added to the View hierarchy.
  • A Composition is created, attached to a Recomposer.
  • The first composition runs your setContent { } block, building the slot table and creating the LayoutNode tree.
  • The LayoutNode tree integrates with the View system's measure/layout/draw pass.
Takeaway: Compose is a small, elegant runtime — a slot table, a Composer, and a snapshot system. Once you understand these three, "advanced" Compose stops feeling mysterious. Reading the kotlinx Compose source becomes routine, not intimidating.

★ Composition vs Recomposition — what they mean, how Google built it, how to skip work

People throw "composition" and "recomposition" around like the same thing. They're not. Understanding the difference unlocks performance work and impresses interviewers.

The vocabulary (clear definitions)

TermWhat it means
Composition (the noun)The in-memory representation of your UI tree at a moment in time. It's a data structure — specifically, a slot table — that records which Composables ran with which parameters, what they remember'd, and how they nest.
Composition (the verb / "composing")The act of running a Composable function for the first time. Adds new entries to the slot table.
RecompositionThe act of running a Composable function again with new inputs. Updates existing entries in the slot table at the same positions.
Initial compositionThe first composition for a given scope — happens once when it enters the tree.
SkippingCompose decides "all inputs unchanged AND stable, no need to re-run the body" and advances the slot table pointer past this Composable's entries without executing.
RecomposeScopeThe smallest unit Compose can re-run independently. By default, every restartable @Composable function creates one.

So: a Composable is composed once when it first appears, and recomposed many times as state changes. Both go through the same compiler-generated code path — the difference is whether the slot table already has entries at this position.

How Google built this — the inheritance from existing computer science

Compose isn't magic from outer space. Every piece of it has roots in older, well-understood technology. Google's contribution was integrating these proven ideas into a coherent UI framework with first-class Kotlin compiler support — not invented-here. Here's the genealogy:

Compose conceptWhere it came from
Slot table (gap buffer array)Text editors since the 1970s. Emacs uses gap buffers for the same insert/delete-in-the-middle efficiency reason. Compose stores its tree as a flat array; the "gap" lets it shift entries when the tree mutates.
Positional memoizationFunctional programming + React's reconciliation idea. React diffs a virtual DOM tree; Compose tracks position in a flat slot table — same goal (identify "this element didn't move"), different data structure.
Snapshot system (MVCC)Multi-Version Concurrency Control from databases since the 1980s — Postgres, Oracle, etc. use MVCC for "consistent reads while writers are mutating." Compose uses the exact same idea for State, so composition can read state on the UI thread while another coroutine mutates it.
Compiler plugin / IR transformationSame Kotlin IR (Intermediate Representation) pipeline that suspend functions go through. Compose Compiler is essentially "suspend-style code transformation, but for UI state." The same Kotlin compiler team that built suspend built Compose Compiler.
State observation (snapshot reads)MobX, Vue's reactivity system, RxSwift Observables, Knockout.js. Same idea: track which code reads which state by intercepting the read.
Modifier chain (decorator pattern)Gang of Four Decorator pattern (1994). Identical structurally to OkHttp's interceptor chain, Servlet filters, middleware in any web framework.
Composer (state machine)Coroutines' Continuation pattern — both are state-machine-driven control flow. Calls like startRestartGroup / endRestartGroup mirror coroutines' $label + resumeWith.
Recomposer (frame batching)Choreographer + VSYNC, which Android has had since API 16. Compose hooks into Choreographer's frame callbacks — when a state change happens, the Recomposer batches all invalidations and applies them at the next frame boundary.
Stability inferenceType system theory — similar in spirit to Rust's borrow checker reasoning about immutability and aliasing.
Single-shot execution modelReact's "render is just a function" idea (2013). Compose pushed it further by making the function literally re-runnable at the granularity of a RecomposeScope.

The clever choice — flat slot table vs virtual DOM tree

The non-obvious decision: Compose uses a flat slot table instead of a virtual DOM tree like React.

  • React approach: each render builds a new tree of plain objects, then diffs against the previous tree. Allocation per render + tree-diff cost.
  • Compose approach: a single flat array (gap buffer) where each slot is a (key, value) pair. Recomposition walks the array and updates in place. Zero allocation per recomposition. Cache-friendly memory access.

Trade-off: Compose can't ask "what would this UI look like with totally new state?" without actually running the code. React can compute "next tree" without committing it. For mobile UI, the Compose trade-off is the right one — way faster on memory-constrained devices.

What's actually inside the slot table for a simple Composable

@Composable
fun Counter() {
    var count by remember { mutableStateOf(0) }
    Text("Clicks: $count")
    Button(onClick = { count++ }) { Text("+") }
}

The slot table contains roughly:

[restart-group: Counter] [remember-group: count] [State<Int>(0)] [restart-group: Text] [parameter: "Clicks: 0"] [LayoutNode(text = ...)] [restart-group: Button] [parameter: lambda#1] [LayoutNode(button = ...)] [restart-group: Text] [parameter: "+"] [LayoutNode(text = ...)] [end-group]

Each entry has a position (index in the array). When recomposition happens, the Composer walks the table from the start of the affected RecomposeScope. If the next entry's parameters match what's stored, it skips. Otherwise it updates in place. Most of recomposition is just walking past entries that haven't changed.

What "smart recomposition" actually means (three optimizations combined)

  1. Targeted invalidation — when state changes, only the smallest enclosing RecomposeScope that read it gets queued for re-execution. Other scopes don't know anything happened.
  2. Stability check at function entry — generated code at the top of every restartable Composable checks the $changed bitmask. If all params are unchanged AND stable, the body is skipped entirely.
  3. Lambda memoization — the Compose Compiler caches lambdas with stable captures so child Composables receive the same lambda reference and stay skippable.

Together: a feed with 100 items where only item #5's data changed re-runs only item #5's restart group. Items 1-4 and 6-100 are skipped — the slot-table walker advances past their entries without execution. That's the whole performance story in one paragraph.

8 common causes of unnecessary recompositions — with code fixes

This is where Compose performance interviews drill. Each pattern: why it happens + the fix.

1. Reading the whole UiState in a parent Composable

Bad:

@Composable
fun Header(vm: MyViewModel) {
    val state by vm.uiState.collectAsStateWithLifecycle()
    Text("Welcome ${state.userName}")        // recomposes on EVERY field change
    LotsOfOtherStuff(state)                  // also recomposes on every field change
}

Why: Header reads the whole state object, so it recomposes whenever any field of state changes — even unrelated ones like scroll position.

Good:

@Composable
fun Header(vm: MyViewModel) {
    val userName by remember(vm) {
        derivedStateOf { vm.uiState.value.userName }
    }
    Text("Welcome $userName")              // recomposes ONLY when userName changes
}

// Or — better — expose individual flows from VM:
val userName: StateFlow<String> = uiState.map { it.userName }.stateIn(...)

2. Inline lambdas creating new instances per recomposition

Bad:

@Composable
fun Parent(items: List<Item>) {
    LazyColumn {
        items(items) { item ->
            ItemRow(item, onClick = { handleClick(item.id) })   // new lambda per recomp
        }
    }
}

Why: The lambda { handleClick(item.id) } captures item — a different instance per row. Each recomposition creates a new lambda; ItemRow sees "different" callback and can't skip.

Good:

@Composable
fun Parent(items: List<Item>, onClick: (String) -> Unit) {
    LazyColumn {
        items(items, key = { it.id }) { item ->
            ItemRow(item, id = item.id, onClick = onClick)      // stable lambda
        }
    }
}

// In ItemRow:
@Composable
fun ItemRow(item: Item, id: String, onClick: (String) -> Unit) {
    Row(Modifier.clickable { onClick(id) }) { ... }             // click closure inline is OK
}                                                                // because clickable's modifier handles it

3. Unstable list / collection parameters

Bad:

@Composable
fun UserList(users: List<User>) { ... }
// kotlin.collections.List is UNSTABLE (could be MutableList).
// UserList always recomposes when its parent does, even if users hasn't changed.

Good:

// Add: implementation("org.jetbrains.kotlinx:kotlinx-collections-immutable:0.3.7")

@Composable
fun UserList(users: ImmutableList<User>) { ... }
// ImmutableList is annotated @Immutable. UserList becomes skippable.

// In ViewModel, expose as ImmutableList:
val users: StateFlow<ImmutableList<User>> = userFlow.map { it.toImmutableList() }
    .stateIn(scope, WhileSubscribed(5_000), persistentListOf())

4. Reading scroll position in the Composition phase

Bad:

@Composable
fun Screen(listState: LazyListState) {
    val position = listState.firstVisibleItemIndex     // changes every scroll tick
    Text("Item: $position")                            // recomposes every scroll pixel
    BigList(listState)
}

Good:

@Composable
fun Screen(listState: LazyListState) {
    val showHeader by remember {
        derivedStateOf { listState.firstVisibleItemIndex > 5 }
    }
    AnimatedVisibility(showHeader) { Header() }              // recomposes only on threshold flip
    BigList(listState)
}

5. Passing State<T> instead of T

Bad:

@Composable
fun NameDisplay(name: State<String>) {
    Text(name.value)                          // reads inside
}

Why: The caller passes the same State reference; the child reads .value inside. Now the child's reads invalidate this scope on every state change. By passing the State down, you've moved the read from a leaf to a parent — inverting what skipping wants.

Good:

@Composable
fun NameDisplay(name: String) {
    Text(name)                                // receives plain value
}

// Caller reads State and passes value:
val name by vm.name.collectAsStateWithLifecycle()
NameDisplay(name)

6. Animations driving high-frequency value reads in Composition

Bad:

@Composable
fun RotatingIcon(angle: Float) {
    Icon(
        modifier = Modifier.rotate(angle),    // reads angle in COMPOSITION phase
        imageVector = Icons.Default.Star,
        contentDescription = null
    )
    // → recomposes EVERY animation frame
}

Good:

@Composable
fun RotatingIcon(angle: Float) {
    Icon(
        modifier = Modifier.graphicsLayer { rotationZ = angle },   // reads in DRAWING phase
        imageVector = Icons.Default.Star,
        contentDescription = null
    )
    // → no recomposition, no relayout, just GPU-side transform per frame
}

7. Index as list key (or no key)

Bad:

LazyColumn {
    itemsIndexed(users) { index, user ->
        UserRow(user)                         // no key
    }
}

Why: Removing user #0 makes Compose think every other user changed (because each one shifted positions). Massive recomposition cascade.

Good:

LazyColumn {
    items(users, key = { it.id }) { user ->
        UserRow(user)                         // stable identity per row
    }
}
// Removing user #0 only animates that one row out; others are untouched.

8. Big data class with one var field destroying stability

Bad:

data class Profile(
    val id: String,
    val name: String,
    var lastSeen: Long                    // var → unstable
)
// Compose marks the whole class unstable; every Composable taking Profile loses skipping.

Good:

@Immutable
data class Profile(
    val id: String,
    val name: String,
    val lastSeen: Long                    // all val
)
// Or, if the property MUST be mutable, model it as State:
@Stable
class ProfileVm {
    var lastSeen by mutableStateOf(0L)
}

How to verify your fixes worked

  1. Layout Inspector → "Show recomposition counts" — interact with the screen; counts should not climb absurdly. Anything > 10× higher than items appearing on screen is a smell.
  2. Compose Compiler metrics (Gradle composeCompiler.metricsDestination) — generated *-composables.txt shows which Composables are skippable and restartable. Goal: every Composable that takes parameters should be both.
  3. Macrobenchmark FrameTimingMetric — measure end-to-end frame time on a release build. Should be under 16.6 ms (60 Hz) or 8.3 ms (120 Hz).

The interview-cheat answer to "How do I avoid unnecessary recompositions?"

Memorize this 5-step answer. Delivers in 30 seconds, demonstrates depth:
  1. "I generate the Compose Compiler stability report — it tells me which Composables are skippable and which aren't, with reasons."
  2. "For unstable parameters (List, Map, classes from other modules), I switch to ImmutableList from kotlinx.collections.immutable or annotate the type @Immutable."
  3. "I defer State reads to the lowest level — read name in a leaf Text, not the whole UiState in a parent Container."
  4. "For animated values, I move reads to the drawing phase via Modifier.graphicsLayer { } instead of value-form modifiers like Modifier.rotate."
  5. "I always provide stable keys in lazy lists — entity ID, never index — and avoid passing inline lambdas that capture different values per row."

7. Side effects — picking the right effect API

Composables must be pure. To do anything that affects the outside world (network call, register a listener, start a timer), use a side-effect API. Compose ties these to the Composable's lifecycle so cleanup is automatic.

The decision tree

produceState(initial, key) { value = ... }Convert a non-Compose async source to State<T>Coroutine cancelled on scope leave / key change
derivedStateOf { computed }Compute a value derived from other State; only recomposes readers when result changesTied to its enclosing remember
rememberUpdatedState(value)Capture latest value in a long-running effect without restarting itReference; no cleanup
 
@Composable fun UserScreen(userId: String) {
    // LaunchedEffect — restart on userId change
    LaunchedEffect(userId) { vm.load(userId) }

    // DisposableEffect — register/unregister
    DisposableEffect(Unit) {
        analytics.startTracking()
        onDispose { analytics.stopTracking() }
    }

    // rememberCoroutineScope — launch from event handlers
    val scope = rememberCoroutineScope()
    Button(onClick = { scope.launch { vm.refresh() } }) { Text("Refresh") }
}
Don't: call LaunchedEffect from inside an onClick. LaunchedEffect is a Composable, not a regular function — it must be in the Composable body. For event-driven launches use rememberCoroutineScope.

The classic rememberUpdatedState use case

@Composable fun Timer(onTimeout: () -> Unit) {
    // Capture latest onTimeout, but don't restart the timer when it changes
    val currentOnTimeout by rememberUpdatedState(onTimeout)
    LaunchedEffect(Unit) {
        delay(5_000)
        currentOnTimeout()           // always uses the latest version
    }
}

8. derivedStateOf — when to use it (and when not)

derivedStateOf wraps a computed value and only emits a new value when the result changes — even if its inputs change frequently.

@Composable fun ScrollAwareFab(listState: LazyListState) {
    // Bad: recomposes every scroll pixel — even when the boolean doesn't flip
    val showButton = listState.firstVisibleItemIndex > 10

    // Good: only recomposes when the boolean actually flips
    val showButton by remember {
        derivedStateOf { listState.firstVisibleItemIndex > 10 }
    }

    if (showButton) FloatingActionButton(...) { Icon(...) }
}
Use derivedStateOf whenDon't use it when
Inputs change far more often than the derived result (scroll position → "show button?")The input changes infrequently — adds allocation + indirection for no gain
The derivation is expensive AND inputs are noisyYou're just transforming one State into another with the same change rate
You want a single flat boolean derived from multiple noisy sourcesYou're inside a non-Composable — derivedStateOf is a runtime construct, can be used outside Composables but typically paired with remember
Common mistake: derivedStateOf without remember is rebuilt on every recomposition — same observation cost as plain reads. Always pair with remember.

9. Stability — why it matters and how to fix unstable types

"Stable" types are types Compose can prove won't change in surprising ways. Stable + unchanged parameters = the Composable can be skipped on recomposition. Unstable parameters = Compose conservatively re-runs the Composable.

What's stable by default

  • Primitives (Int, Long, Boolean, Float, Double, Char) and Strings
  • Function types (lambdas) — but only if their captures are stable
  • State<T> and MutableState<T>
  • Types annotated @Stable or @Immutable
  • data classes whose properties are all stable types

What's not stable by default

  • List<T>, Set<T>, Map<K, V> from kotlin.collections — they could be MutableList
  • Interfaces (Compose can't see the implementation; assumes worst case)
  • Classes with var properties
  • Classes from other modules (compiler can't infer stability without seeing the source)

The two annotations

@Immutable
data class User(val id: String, val name: String)
// Promise: properties never change after construction. Compose treats as fully stable.

@Stable
class CounterStore {
    var count by mutableStateOf(0)
}
// Promise: properties may change, but I'll notify Compose via State. Compose treats as stable.

Strong skipping mode (Compose Compiler 1.5.4+)

Even unstable parameters can now be skippable if they pass an equals() check. Practical effect: most stability bugs disappear in Compose 1.6+. Still, mark types @Immutable when you can — it's clearer and handles cases where equals is O(N).

How to inspect stability

Add the Compose Compiler metrics flag to your Gradle config:

composeCompiler {
    metricsDestination = layout.buildDirectory.dir("compose-metrics")
    reportsDestination = layout.buildDirectory.dir("compose-reports")
}

After a release build, the reports directory contains per-module *-classes.txt (stability of every class) and *-composables.txt (skippable / restartable status of every Composable). Best perf-debugging tool you can ship to your team.

Common stability fixes

  • Use kotlinx.collections.immutable — ImmutableList<T>, PersistentList<T> are @Immutable.
  • Make data class properties val, not var.
  • Annotate domain models @Immutable.
  • For interface parameters, mark them @Stable on the interface itself or replace with sealed classes.

10. key() and list itemKey — identity in lists

By default, Compose identifies items by position in the call tree. For lists where items can be added, removed, or reordered, you must give Compose a stable identity, or it'll associate state with the wrong item.

LazyColumn {
    items(
        items = users,
        key = { user -> user.id }              // stable identity
    ) { user ->
        UserRow(user)
    }
}

// Inside a non-list Composable: key() block
users.forEach { user ->
    key(user.id) {                              // scoped identity
        UserRow(user)
    }
}

Symptoms of missing keys:

  • State stuck on the wrong item after add/remove (e.g., a checkbox state moves)
  • Janky animations that look like rebuilds rather than moves
  • rememberSaveable values appearing on the wrong rows
Don't: use the index as the key. That's the same as no key — defeats the whole purpose. Don't use the entire object either; equals must be cheap and identity-meaningful.

11. Modifiers — order matters; Modifier.Node for performance

A Modifier is an immutable, ordered chain of behaviors applied to a Composable: padding, click, background, etc. Order is meaningful — each modifier wraps the next.

// 16 dp padding, then background → padding outside the color
Box(Modifier.padding(16.dp).background(Color.Red))

// background, then padding → padding inside the color
Box(Modifier.background(Color.Red).padding(16.dp))

Common modifier reading order

  1. Size (fillMaxWidth, size, height) defines the constraints
  2. Padding adds inset
  3. Background paints behind contents
  4. Border draws on top of background
  5. Click areas use the modified bounds

Custom modifiers — use Modifier.Node

// Modern (recommended) — Modifier.Node + ModifierNodeElement
private class MyHighlightNode : Modifier.Node(), DrawModifierNode {
    override fun ContentDrawScope.draw() {
        drawRect(Color.Yellow.copy(alpha = 0.3f))
        drawContent()
    }
}

private data object MyHighlightElement : ModifierNodeElement<MyHighlightNode>() {
    override fun create() = MyHighlightNode()
    override fun update(node: MyHighlightNode) {}
}

fun Modifier.highlight() = this then MyHighlightElement

Why Modifier.Node over the older composed { }:

  • Reusable across recompositions — fewer allocations
  • Doesn't trigger recomposition just because the modifier was rebuilt
  • Can hook into the layout, draw, focus, pointer-input phases via specialized interfaces (DrawModifierNode, LayoutModifierNode, etc.)
Don't: use composed { } for new code. It's flagged as deprecated in newer Compose versions and creates an extra Composable per call site.

12. The three phases — Composition, Layout, Drawing

Compose runs three phases per frame, top-down. Understanding which phase a State read happens in determines what gets invalidated when the State changes.

PhaseWhat happensState read here invalidates
CompositionRun @Composable functions, build the tree (slot table + LayoutNodes)Composition + Layout + Drawing
LayoutMeasure each node (with constraints from parent), then place childrenLayout + Drawing
DrawingPaint pixels into the bufferDrawing only

Defer state reads to the latest possible phase

// Bad — reads scroll value in Composition phase → recomposes on every scroll pixel
Text("Offset: ${scrollState.value}")

// Good — reads in Layout phase via lambda → only relayouts, no recomposition
Box(Modifier.offset { IntOffset(0, scrollState.value) })

// Even better for transforms — reads in Drawing phase → only redraws
Box(Modifier.graphicsLayer { translationY = scrollState.value.toFloat() })

Modifiers that take a lambda (Modifier.offset { }, Modifier.graphicsLayer { }, Modifier.drawBehind { }) defer execution to a later phase. Reading State inside the lambda doesn't trigger recomposition — only the relevant later-phase invalidation.

13. Custom Layout and SubcomposeLayout

For most cases, built-in containers (Box, Row, Column, LazyColumn) are enough. When you need custom measurement/placement, drop to Layout.

@Composable
fun SpacedColumn(spacing: Dp, content: @Composable () -> Unit) {
    Layout(content = content) { measurables, constraints ->
        val placeables = measurables.map { it.measure(constraints) }
        val spacingPx = spacing.roundToPx()
        val totalH = placeables.sumOf { it.height } + spacingPx * (placeables.size - 1)
        layout(constraints.maxWidth, totalH) {
            var y = 0
            placeables.forEach { it.placeRelative(0, y); y += it.height + spacingPx }
        }
    }
}

SubcomposeLayout — when you need measurement to influence composition

Used internally by Scaffold, BoxWithConstraints, LazyColumn. Lets you compose children based on measurement results from other children.

SubcomposeLayout { constraints ->
    val headerPlaceable = subcompose("header") { Header() }
        .first().measure(constraints)
    val remaining = constraints.copy(maxHeight = constraints.maxHeight - headerPlaceable.height)
    val bodyPlaceable = subcompose("body") { Body() }
        .first().measure(remaining)
    layout(constraints.maxWidth, constraints.maxHeight) {
        headerPlaceable.placeRelative(0, 0)
        bodyPlaceable.placeRelative(0, headerPlaceable.height)
    }
}
Don't: reach for SubcomposeLayout if a regular Layout works. SubcomposeLayout composes during measure, which disables intrinsics and adds significant cost.

14. CompositionLocal — implicit values down the tree

A way to pass values implicitly through the Composable tree without threading them through every parameter. Built-in examples: LocalContext, LocalDensity, LocalConfiguration, MaterialTheme.colorScheme.

val LocalAppUser = compositionLocalOf<User> { error("No User provided") }

@Composable fun App() {
    val currentUser = vm.user.collectAsStateWithLifecycle().value
    CompositionLocalProvider(LocalAppUser provides currentUser) {
        Screen()
    }
}

@Composable fun Avatar() {
    val user = LocalAppUser.current      // implicit lookup
    Image(user.avatar, contentDescription = user.name)
}

Two flavors

  • compositionLocalOf — only direct readers recompose when the value changes (more granular)
  • staticCompositionLocalOf — entire subtree recomposes on change. Cheaper at runtime if the value rarely changes (Theme, Density)

15. Lists — LazyColumn and friends

Lazy components only compose visible items + a small pre-fetch window. They handle thousands of items efficiently.

Performance rules for lists

  • Always provide key. Always.
  • Provide contentType when items can have different shapes — Compose recycles by type.
  • Don't read listState.firstVisibleItemIndex in Composition unless wrapped in derivedStateOf (§8).
  • Use animateItemPlacement() on item Composables for free reorder animations.
  • Hoist heavy state outside the item Composable; the item function should be lightweight.

24. Performance — finding and fixing recompositions

How to find offending recompositions

  1. Layout Inspector → "Show recomposition counts" — see counts per Composable as you interact.
  2. Compose Compiler metrics report (§9) — per-Composable skippable / restartable info; per-class stability.
  3. Macrobenchmark + FrameTimingMetric — measure end-to-end frame timing on a release build.
  4. Perfetto trace — low-level system trace; see scheduling and main-thread work.

Common fixes

  • Stable parameters: replace List<T> with ImmutableList<T> from kotlinx.collections.immutable.
  • Defer State reads to later phases: use lambda-modifiers (Modifier.offset { }, graphicsLayer { }) instead of value-modifiers.
  • Wrap derived values: derivedStateOf for computed booleans on noisy inputs.
  • Cache lambdas: in scroll callbacks, val onClick = remember { { vm.click() } } avoids new Function instances per frame.
  • Stable keys for lists: never index, always entity ID.
  • Ship a Baseline Profile: pre-AOT-compile your hot paths. Compose-heavy apps see 20-40% cold-start improvement.
  • Use primitive State: mutableIntStateOf over mutableStateOf(0) — avoids boxing.
Don't profile debug builds. Compose generates lots of extra debug-only allocations. Always profile a release build with R8 + Baseline Profile installed.

25. Do's and Don'ts — comprehensive checklist

DO
  • Make UI a function of state. UI = f(state).
  • Hoist state to the lowest common ancestor of Composables that need it.
  • Use collectAsStateWithLifecycle for Flow in Compose.
  • Provide stable keys for every list.
  • Mark domain models @Immutable.
  • Use derivedStateOf for noisy inputs producing rare boolean outputs.
  • Use graphicsLayer { } lambda for animated transforms.
  • Use Modifier.testTag for stable test selectors.
  • Run release build + Baseline Profile when measuring perf.
  • Provide contentDescription for meaningful images; null for decorative.
  • Use rememberSaveable for UI state that must survive process death.
  • Pair side effects with the right effect API (LaunchedEffect, DisposableEffect).
DON'T
  • Mutate plain var from inside a Composable expecting recomposition.
  • Run network calls or I/O directly in the Composable body.
  • Use collectAsState() in production — wastes battery in background.
  • Use the index as a list key.
  • Call LaunchedEffect from an onClick.
  • Forget remember around derivedStateOf / mutableStateOf.
  • Pass mutable List<T> to Composables and expect skipping.
  • Read scrollState.value in Composition without derivedStateOf.
  • Use SubcomposeLayout when a regular Layout works.
  • Mass-apply safeDrawing at the root over a colored TopAppBar.
  • Profile debug builds for performance work.
  • Ignore the Compose Compiler metrics report — it's the truth.

27. One-page recall card

The 30-second mental model

Compose = declarative UI. UI = f(state). Compose Compiler rewrites every @Composable to inject a Composer + change-tracking bitmask + restart group. Composer maintains the slot table. Snapshot system tracks State reads per RecomposeScope. State change → invalidates the smallest enclosing scope → next frame, that scope re-runs.

State

  • mutableStateOf, mutableIntStateOf for primitives
  • remember { } for in-composition; rememberSaveable { } for process death
  • derivedStateOf { } + remember for noisy inputs → rare output
  • State hoisting: pass (value, onChange), hoist to lowest common ancestor

Side effects

  • LaunchedEffect(key) — coroutine, restart on key change
  • DisposableEffect(key) — register/unregister with onDispose
  • SideEffect — publish to non-Compose code
  • rememberCoroutineScope() — launch from event handlers
  • produceState — async source → State
  • rememberUpdatedState — capture latest in long-running effect without restart

Performance

  • Stable parameters → skipping; mark with @Immutable
  • ImmutableList<T> over List<T>
  • Defer State reads to Layout/Drawing via lambda-modifiers
  • graphicsLayer { } for animated transforms
  • Stable keys in lists (entity ID, never index)
  • Compose Compiler metrics report = ground truth
  • Baseline Profile for cold start

Internals (the engine)

  • Compiler injects Composer + $changed bitmask
  • Body wrapped in startRestartGroup / endRestartGroup + updateScope
  • Slot table = flat gap-buffer array of all groups, parameters, remembered values
  • Positional memoization = remember uses call-site position as the key
  • Snapshot system = MVCC; read tracking via thread-local SnapshotObserver
  • RecomposeScope = smallest re-runnable unit; State change invalidates the scope that read it
  • Recomposer = singleton coroutine on AndroidUiDispatcher; batches per frame

Avoid

  • collectAsState() (use lifecycle variant)
  • Index as list key
  • Plain var for UI state
  • Side effects in Composable body
  • LaunchedEffect inside onClick
  • Profiling debug builds
  • SubcomposeLayout when Layout works
  • Mutable List<T> as Composable parameter
